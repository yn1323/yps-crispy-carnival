# ディレクトリ構造とコンポーネント責務分離

## ステータス
承認済み

## 背景

プロジェクトの開発を進める上で、コンポーネントの責務とディレクトリ構造を明確化する必要があった。特に以下の点について明確な方針が必要だった：

- `/routes`、`/pages`、`/features` 各層の責務範囲
- データ取得のタイミングと場所
- mutation系の処理をどこに配置するか
- 状態別の条件分岐をどこで実行するか
- 開発スピードと保守性のバランス

## 決定事項

### 呼び出し関係性

```
/routes → /pages → /features
```

各層は明確な責務を持ち、上位層から下位層を呼び出す一方向の依存関係とする。

### /routes 配下の責務

- **役割**: ルーティング定義とページコンポーネントの呼び出し
- **構造**: 
  - ディレクトリ名 = URLパス
  - ファイルは必ず `index.tsx`
- **実装内容**: `<Animation>` でページコンポーネントをラップして呼び出すだけ
- **禁止事項**: ロジック、データ取得、状態管理を含めない

### /pages 配下の責務

- **役割**: 初回描画に必要なデータ取得とページレベルの状態管理
- **主な処理**:
  - Query系データ取得（React Query等）
  - ドメイン情報取得
  - **初回描画で必要な全てのデータを一括取得（B案採用）**
  - **状態別の条件分岐（ローディング、空、エラー等）**
- **制約**:
  - mutation系は含まない
  - UI実装は最小限
- **必須ファイル**: `index.stories.tsx` + `index.tsx` をセット

### /features 配下の責務

- **役割**: UI実装とインタラクション処理
- **主な処理**:
  - レイアウト構築
  - UI処理（クリック、フォーム入力等）
  - mutation系処理（データ更新、削除等）
- **設計方針**:
  - **mutation系はコンポーネントと同じディレクトリに配置（A案採用: スピード重視）**
  - **状態別コンポーネントを同じファイルから複数export**
    - 例: `ShiftList`（通常）、`ShiftListEmpty`（空）、`ShiftListLoading`（ローディング）
- **必須ファイル**: `index.stories.tsx` + `index.tsx` をセット

### 設計原則

#### 1. スピード重視
- コンポーネント分割より実装速度を優先
- 共通化は実際に必要になってから
- 過度な抽象化は避ける

#### 2. コロケーション（Locality of Behavior）
- 関連するものは同じディレクトリに配置
- mutation系は対応するコンポーネントと同じディレクトリ
- 依存関係が明確で変更影響範囲が限定される

#### 3. 初回データ取得の最適化
- `/pages` で可能な限り全てのデータを取得
- 並列データ取得によるパフォーマンス最適化
- ローディング状態の一元管理

#### 4. 条件分岐の責務分離
- `/pages`: データの状態判定と適切なコンポーネント選択
- `/features`: 状態に応じた複数のコンポーネントをexport
- 責務が明確で、テストが容易

## 具体例

### /routes の実装例
```tsx
// routes/shifts/index.tsx
import { ShiftsPage } from '@/components/pages/ShiftsPage';
import { Animation } from '@/components/ui/Animation';

export const Route = () => {
  return (
    <Animation>
      <ShiftsPage />
    </Animation>
  );
};
```

### /pages の実装例
```tsx
// pages/ShiftsPage/index.tsx
import { useShifts } from '@/hooks/useShifts';
import { ShiftList, ShiftListEmpty, ShiftListLoading } from '@/components/features/ShiftList';

export const ShiftsPage = () => {
  const { data: shifts, isLoading, error } = useShifts();
  
  if (isLoading) return <ShiftListLoading />;
  if (error) return <ErrorState error={error} />;
  if (!shifts || shifts.length === 0) return <ShiftListEmpty />;
  
  return <ShiftList shifts={shifts} />;
};
```

### /features の実装例
```tsx
// features/ShiftList/index.tsx
import { useDeleteShift } from './mutations';

// 通常状態
export const ShiftList = ({ shifts }) => {
  const { mutate: deleteShift } = useDeleteShift();
  
  return (
    <Box>
      {shifts.map(shift => (
        <ShiftItem 
          key={shift.id} 
          shift={shift}
          onDelete={() => deleteShift(shift.id)}
        />
      ))}
    </Box>
  );
};

// 空状態
export const ShiftListEmpty = () => {
  return <EmptyState message="シフトがありません" />;
};

// ローディング状態
export const ShiftListLoading = () => {
  return <Spinner />;
};
```

### mutation の実装例
```tsx
// features/ShiftList/mutations.ts
import { useMutation } from '@tanstack/react-query';

export const useDeleteShift = () => {
  return useMutation({
    mutationFn: (shiftId: string) => api.deleteShift(shiftId),
    onSuccess: () => {
      queryClient.invalidateQueries(['shifts']);
    }
  });
};
```

## 影響

### ポジティブな影響

1. **開発速度の向上**
   - 責務が明確で迷わない
   - コンポーネント配置の判断が容易
   - mutation系の配置場所が一貫

2. **保守性の向上**
   - 変更影響範囲が限定的
   - コロケーションによる理解のしやすさ
   - 状態別コンポーネントでテストが容易

3. **パフォーマンスの最適化**
   - 初回データの並列取得
   - 不要な再取得の防止
   - ローディング状態の一元管理

4. **コードの一貫性**
   - 全てのページで同じパターン
   - 新規開発時の迷いが減少
   - レビューが容易

### ネガティブな影響・トレードオフ

1. **初期の柔軟性の制約**
   - mutation系をコンポーネントと同じ場所に配置するため、共通化のタイミングを見極める必要がある
   - 初回で全データ取得するため、大量データの場合はページネーションやフィルタリングの設計が必要

2. **学習コスト**
   - 新規参加者はこの構造を理解する必要がある
   - ただし、ADRとして文書化されているため、理解しやすい

3. **リファクタリングコスト**
   - 後から共通化が必要になった場合、ファイル移動が発生する
   - ただし、スピード重視の方針により、初期段階では許容

## 備考

この設計は「スピード重視」と「保守性」のバランスを取ったものである。実際の開発を進める中で問題が発覚した場合は、適宜見直しを行う。

特に以下の点については継続的にモニタリングする：
- mutation系の共通化の必要性
- 初回データ取得のパフォーマンス影響
- 状態別コンポーネントの増加による複雑性
