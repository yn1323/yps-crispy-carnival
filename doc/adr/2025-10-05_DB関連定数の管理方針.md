# DB関連定数の管理方針

**ステータス**: 承認済み  
**日付**: 2025-10-05  
**決定者**: 開発チーム

---

## 背景

店舗管理機能の実装において、以下のDB関連定数をフロントエンド（`src/`）とバックエンド（`convex/`）の両方で使用する必要が発生した：

- `SHOP_TIME_UNIT`: シフト時間単位の選択肢 `[1, 5, 10, 15, 30, 60]`
- `SHOP_SUBMIT_FREQUENCY`: シフト提出頻度の選択肢 `["1w", "2w", "1m"]`
- `SHOP_MIN_LENGTH`, `SHOP_MAX_LENGTH`: 店舗名の文字数制限

### 技術的制約

Convexは独立したTypeScriptプロジェクトとして動作する：

```
convex/
├── tsconfig.json        # 独立したTypeScript設定
├── shop.ts             # サーバーサイドコード
└── constants.ts        # サーバー側定数

src/
├── constants/
│   └── validations.ts  # クライアント側定数
└── components/         # フロントエンドコード
```

**重要な制約:**
- ❌ `convex/` → `src/` のimportは**不可能**（Convexのビルドシステムの制約）
- ✅ `src/` → `convex/` のimportは**可能**（tsconfig.jsonに`@/convex/*`パスエイリアスあり）

### 検討した選択肢

#### 選択肢1: 両方で定数を定義（重複）
```typescript
// convex/shop.ts
const SHOP_TIME_UNIT = [1, 5, 10, 15, 30, 60] as const;

// src/constants/validations.ts
export const SHOP_TIME_UNIT = [1, 5, 10, 15, 30, 60] as const;
```

**メリット:**
- シンプル

**デメリット:**
- ❌ 定数の重複（保守性が低い）
- ❌ 変更時に2箇所修正が必要
- ❌ 同期漏れのリスク

#### 選択肢2: convex/tsconfig.jsonのincludeを変更
```json
"include": ["./**/*", "../src/**/*"]
```

**メリット:**
- 一見すると統合できそう

**デメリット:**
- ❌ Convexのビルドシステムと衝突
- ❌ `src/`のコードはConvexサーバーにデプロイされない
- ❌ ランタイムでimportが失敗する
- ❌ サーバーがクライアントコードに依存する設計になる（アンチパターン）

#### 選択肢3: convex/constants.tsを唯一の真実の源（Single Source of Truth）にする ✅

```typescript
// convex/constants.ts（定義元）
export const SHOP_TIME_UNIT = [1, 5, 10, 15, 30, 60] as const;
export const SHOP_SUBMIT_FREQUENCY = ["1w", "2w", "1m"] as const;

// src/constants/validations.ts（参照側）
export { 
  SHOP_TIME_UNIT, 
  SHOP_SUBMIT_FREQUENCY 
} from "@/convex/constants";
```

**メリット:**
- ✅ **Single Source of Truth**（定数の重複がゼロ）
- ✅ 変更は1箇所だけでOK
- ✅ DB関連の定数はサーバー側が管理するのが自然
- ✅ Convexの思想に沿っている（クライアントがサーバーをimportするのは普通）
- ✅ tsconfig.jsonの変更不要（既存のパスエイリアスを活用）
- ✅ 型の一貫性が保たれる

**デメリット:**
- クライアント側がサーバー側のコードに依存する（ただしConvexでは一般的）

## 決定事項

**選択肢3を採用する：`convex/constants.ts`を唯一の真実の源とする**

### 実装方針

1. **convex/constants.ts を作成**
   - 全てのDB関連定数を定義
   - 型定義も含める

2. **src/constants/validations.ts を修正**
   - DB関連定数は`convex/constants.ts`からre-export
   - フロントエンドのみの定数（PASSWORD_MIN_LENGTH等）はそのまま残す

3. **convex/shop.ts を修正**
   - インライン定数を削除
   - `import { SHOP_TIME_UNIT, SHOP_SUBMIT_FREQUENCY } from "./constants"`

### 定数の分類基準

**convex/constants.ts に配置するもの:**
- ✅ DB関連の定数（テーブル定義に関わるもの）
- ✅ バリデーションに使用する選択肢
- ✅ フロントエンド・バックエンド両方で使用するもの

**src/constants/validations.ts に残すもの:**
- フロントエンドのみで使用する定数
- UI表示のみに関わる定数

### コード例

```typescript
// convex/constants.ts
export const SHOP_TIME_UNIT = [1, 5, 10, 15, 30, 60] as const;
export type ShopTimeUnitType = (typeof SHOP_TIME_UNIT)[number];

export const SHOP_SUBMIT_FREQUENCY = ["1w", "2w", "1m"] as const;
export type ShopSubmitFrequencyType = (typeof SHOP_SUBMIT_FREQUENCY)[number];

export const SHOP_MIN_LENGTH = 2;
export const SHOP_MAX_LENGTH = 50;
```

```typescript
// src/constants/validations.ts
// DB関連の定数はconvex/constants.tsから参照（Single Source of Truth）
export { 
  SHOP_TIME_UNIT, 
  SHOP_SUBMIT_FREQUENCY,
  SHOP_MIN_LENGTH,
  SHOP_MAX_LENGTH,
  type ShopTimeUnitType,
  type ShopSubmitFrequencyType 
} from "@/convex/constants";

// フロントエンドのみの定数
export const PASSWORD_MIN_LENGTH = 8;
export const PASSWORD_MAX_LENGTH = 24;

export const USER_MIN_LENGTH = 2;
export const USER_MAX_LENGTH = 20;
```

```typescript
// convex/shop.ts
import { SHOP_TIME_UNIT, SHOP_SUBMIT_FREQUENCY } from "./constants";

export const createShop = mutation({
  handler: async (ctx, args) => {
    // バリデーション: timeUnit
    if (!SHOP_TIME_UNIT.includes(args.timeUnit as typeof SHOP_TIME_UNIT[number])) {
      throw new ConvexError({
        message: "シフト時間単位が不正です",
        code: "INVALID_TIME_UNIT",
      });
    }
    // ...
  },
});
```

## 影響

### ポジティブな影響

1. **保守性の向上**
   - 定数の変更が1箇所で完結
   - 同期漏れのリスクがゼロ

2. **型安全性の向上**
   - TypeScriptの型推論が一貫して機能
   - フロントエンドとバックエンドで型の不整合が発生しない

3. **設計の明確化**
   - DB関連の定数はサーバー側が管理する責務が明確
   - クライアント側は参照するだけ

4. **Convexの思想に沿った設計**
   - クライアントがサーバーのAPIや定数をimportするのはConvexの標準的な使い方

### ネガティブな影響

1. **依存関係の方向**
   - クライアント側がサーバー側のコードに依存
   - ただし、Convexでは`api.xxx.xxx`のように既に依存しているため、実質的な影響は小さい

2. **学習コスト**
   - 新規参加者がこの設計方針を理解する必要がある
   - ドキュメント（このADR）で明示することで緩和

### 移行作業

1. `convex/constants.ts` を新規作成
2. `convex/shop.ts` のインライン定数を削除してimport
3. `src/constants/validations.ts` を修正してre-export
4. 既存のimportは変更不要（パスは同じなので）
