## 2. 目的

### 根本目的

**AIO経由などで「シフトリってどんなツール？」と興味を持った店長が、資料請求や会員登録を挟まずに、その場で使用感を確かめられる状態を作る。**

### ゼロ負担哲学の拡張

シフトリのコア価値「スタッフへのゼロ負担」の本当の意味は
「**店長がスタッフにアプリ登録や新ツールの使い方説明を促さなくていい**」
であり、店長の心理的・調整コスト負担を消すことが本質。

これを**検討中の店長にも適用する**。

| 対象 | ゼロ負担の形 |
|---|---|
| スタッフ | アカウント登録不要、メールのリンクから提出 |
| 店長（導入後） | 調整コストなくシフトが集まる |
| **店長（検討中）** | **資料請求不要、即触れる** |

「誰に対してもゼロ負担」という一貫した哲学を、競合（Airシフト、ジョブカン等の資料請求・商談フロー）との差別化軸にする。

### 目的の解像度

- 機能説明の網羅ではなく「使えそう」という安心感の提供
- 店長が**自分のペース**で**自分の手で**確認できること
- シフトリの世界観（UI・操作感・ゼロ負担）を**30秒〜2分で**体感できること

## 3. 設計判断の経緯

### 3-1. インタラクティブデモ vs サンドボックス

議論の中で以下2方式を比較した。

| 観点 | インタラクティブデモ | サンドボックス（採用） |
|---|---|---|
| ユーザーの自由度 | 低（決まった道だけ） | 高（自由に探索） |
| 実装の発想 | 演出動画をインタラクティブ化 | 本物のアプリからDBを抜いただけ |
| 「本物感」 | 演出で作る | 本物そのもの |
| コード流用 | 新規実装多め | 既存コンポーネント流用 |
| 向く相手 | 短時間で判断したい人 | 触らないと納得しない人 |
| 田中さんペルソナ適合 | 普通 | 高 |
| シフトリ哲学との整合 | 弱 | 強（「すぐ触れる」） |

**選択**: サンドボックス（DBなし実画面デモ）

**理由**:
- 小規模店舗向けSaaSの意思決定者は現場担当者＝即断できる、触って判断する
- シフトリの哲学「誰に対してもゼロ負担」「すぐ触れる」と整合する
- 既存コンポーネント流用で実装コストを抑えられる

### 3-2. 視点の選択

**採用**: 店長視点のみ

**理由**:
- 「スタッフへのゼロ負担」の本質は「店長が楽」なので、店長視点で伝わる
- スタッフ側の楽さは「メール見るだけ」という事実をLP本文で明示すれば補える
- 両視点を実装すると大規模化、本業並行では非現実的
- 店長が自分で触ることで「誰でも使えそう」という安心感が得られる

スタッフ側の体験については、ツアー終盤に「スタッフにはこういうメールが届く」「提出画面はこんな感じ」の形でチラ見せする。

### 3-3. データ層の技術選択

**採用**: localStorage 完結、Convex / Clerk には一切触らない

**理由**:
- Convex に触ると認証・データ分離・クリーンアップ・悪用防止の設計が泥沼化
- localStorage なら完全にフロントエンドだけで完結
- リロードで初期化する仕様なら永続化の設計不要
- 本番データと完全分離できるので事故リスクゼロ

### 3-4. 完全放流 vs 導線つき

**採用**: 導線つきサンドボックス（本物UI + 薄いストーリー誘導ツアー）

**理由**:
- 完全放流だと「何をすれば『お！』となるか」が伝わらず離脱する
- 完全ツアーだと「自由に触れない」不満が出る
- ストーリー仕立てで操作を誘導しつつ、いつでも自由に触れるハイブリッドが両立点

### 3-5. ツアーライブラリの選定

**採用**: driver.js

**理由**:
- TypeScript完全対応、型が効く
- 軽量（gzip ~5KB）、フレームワーク非依存
- React 19 の breaking change に振り回されない
- 最近もアクティブに更新されている
- MIT、無料
- イベント駆動の進行制御が素直に書ける（`driver.moveNext()` を任意タイミングで呼べる）
- 将来 #341 ツアーガイドにも流用可能

候補として react-joyride も検討したが、更新頻度が落ちている点で却下した。

## 4. 機能仕様

### 4-1. 公開URL

- `/demo` — 店長視点のサンドボックス本体
- `/shifts/submit` 配下の**専用デモトークン用URL** — スタッフ提出画面のデモ閲覧用（詳細は後述）

### 4-2. 認証・データ

- 認証不要（誰でもアクセス可能）
- データは localStorage に保存、キー prefix は `shiftori:demo:v1:*`
- **リロードで初期状態に戻る**（固定データから毎回スタート）
- 初期データは `src/demo/seed.ts` 等に定義

### 4-3. 初期データ（シード）

リアリティある最小構成。

- 店舗名: `居酒屋シフトリ`
- スタッフ: 3〜5人程度（姓のみ。田中、山田、佐藤 など）
- シフト期間: 現在の週 + 来週の2週間
- シフト希望: **全スタッフ分が揃った状態で開始**（空っぽより「もう動いてる」感が強い）
- ポジション: ホール / キッチン 程度

### 4-4. 実装対象機能

シフトボード中心、ただしツアーで誘導する範囲に絞る。

- シフトボード閲覧（ガントチャート）
- シフト追加
- シフト編集（ドラッグ&ドロップによる時間変更を含む）
- シフト削除
- 「確定」ボタン → メール送信演出（実際には送らない）

**スコープ外**:
- スタッフの追加・削除（デモでは初期データで固定）
- 実際のメール送信
- マジックリンクの実発行
- モバイル完全最適化（最低限崩れない程度）
- 複数タブ間でのデータ同期

## 5. ツアーフロー詳細

### 5-1. ステップ構成（全9ステップ）

```
【OPEN】/demo に到達 → driver.js がツアーを自動開始

STEP 1: 導入
  ・ シフトボード表示（スタッフ希望が全員分揃った状態）
  ・ ガイド: 「スタッフ全員が希望を出してくれました」
           「あなたがやったのは『シフトリからメール来るから出してね』とお願いしただけ」

STEP 2: ドラッグ編集
  ・ ガイド: 「田中さんの18-22時のシフト、19-23時にずらしてみて」
  ・ 進行: ドラッグ完了イベントで driver.moveNext()

STEP 3: 時間編集
  ・ ガイド: 「19時にシフトが被りすぎ！佐藤さんの時間を調整してみて」
  ・ 進行: 編集完了イベントで driver.moveNext()

STEP 4: 削除
  ・ ガイド: 「山田さんはこの日お休み希望だったみたい、削除してみて」
  ・ 進行: 削除完了イベントで driver.moveNext()

STEP 5: 確定促し
  ・ ガイド: 「いい感じのシフトになりましたね」
           「確定ボタンを押してスタッフに伝えましょう」
  ・ 進行: 確定ボタンクリックで driver.moveNext()

STEP 6: メール送信演出
  ・ 「メール送信中...✈️」→「送信完了」
  ・ ガイド: 「スタッフにこんなメールが届きます」
  ・ メールプレビュー表示（文面のモック）

STEP 7: スタッフ提出画面チラ見せ
  ・ ガイド: 「ちなみにスタッフの提出画面はこんな感じ」
  ・ 専用URL提示（別タブで開けるリンク、詳細は第6章）

STEP 8: 振り返り
  ・ ガイド: 「店長のあなたがやったこと」
  ・ （文言は別途詰める、候補は第7章）

STEP 9: CTA
  ・ 吹き出し内に「あなたのお店で始める」ボタン
  ・ クリックで `/sign-up` へ遷移
```

### 5-2. 判定ポリシー

**ゆるい判定**を採用。

- 「19-23時にドラッグ」と指示しても、20-24時にドラッグしても進む
- ドラッグ・編集・削除の**アクションが発生したこと自体**を進行条件にする
- 厳密な正解チェックはしない（離脱リスクを優先回避）

### 5-3. ツアー制御

- **開始タイミング**: `/demo` 到達時に自動スタート（身構える時間を作らない）
- **進行スタイル**: 半強制（自動開始）+ スキップあり（ユーザーが任意で終了可能）
- **スキップボタン文言**: 後日詰める。候補は「あとで自由に触る」「ガイドなしで試す」など
- **スキップ時の挙動**: 状態維持のまま自由モードへ（リセットしない）
- **完走後の挙動**: そのまま自由に触れる（最初に戻らない）

### 5-4. CTA配置

- **ツアー中**: 表示しない（ツアーに集中させる）
- **ツアー終了後**: ステップ9の吹き出し内にのみ表示
- **スキップ後**: 吹き出し表示なし（自由に触って気が向いたら登録、という流れ）

常時画面下固定のCTAは**採用しない**。nataniさんの「情報は削る方向」の指針に従う。

### 5-5. デモ明示バナー

**採用しない**。理由:

- スタッフ名がダミー（田中、山田、佐藤）で見れば分かる
- 店舗名「居酒屋シフトリ」も明らかにデモ用
- 没入感を削がない
- 「情報は削る」「被りは許さない」の指針に従う

## 6. スタッフ提出画面デモ（拡張スコープ）

### 6-1. 方針

スタッフ提出画面用の**専用デモURL**を用意する。既存の提出画面コンポーネントを `demoMode` プロパティなどで分岐させ、データ層を localStorage 経由に差し替える。

### 6-2. なぜ画像ではなく実画面か

選択肢として以下を比較:

- 案X: 画像（スクショ）を表示するだけ
- 案Y: デモ用トークンで本物の画面にアクセス可能

採用: **案Y**。

**理由**: シフトリのコンポーネントは疎結合に設計されており、提出画面コンポーネントに `demoMode` を渡すだけで流用可能なはず。コンポーネント思考の設計を活かす。実装コストは案Xより増えるが、「URL踏んだら本物の画面が見える」体験のほうがシフトリの哲学（すぐ触れる・誠実）と整合する。

### 6-3. URL設計の候補

実装時に以下のいずれかを選ぶ。

- `/shifts/submit?token=demo` — 既存URLと同形式、固定デモトークンで分岐
- `/shifts/submit/demo` — 専用パス
- `/demo/staff-submit` — デモ配下に寄せる

どの形でも機能は同じ。既存の提出画面ロジックと干渉しないことが最優先。

### 6-4. スタッフ画面でできること

- 架空のシフト期間・店舗情報が表示される
- 希望時間を入力できる（localStorage 保存、リロードで消える）
- 「提出」ボタンは押せる（完了画面への遷移を試せる、ただしメールは飛ばない）

### 6-5. ツアーからの導線

ステップ7（チラ見せ）で、このデモURLへのリンクを提示。別タブで開く（`target="_blank"` `rel="noopener noreferrer"`）。

## 8. 技術仕様

### 8-1. スタック

既存のシフトリの技術スタックを踏襲。

- TypeScript
- React 19
- Vite
- TanStack Router（ファイルベースルーティング）
- Chakra UI v3
- Jotai（既存の状態管理）
- driver.js（新規導入）

### 8-2. 既存コンポーネントの流用戦略

作業開始時に、既存のデータアクセス層の抽象度を確認する。

- **Good**: Convex の `useQuery` / `useMutation` がカスタム hook や Provider に集約されている → デモ版 hook を作って Context で差し替え
- **Bad**: コンポーネント内に直接散らばっている → 本タスク内では最小限の抽象化にとどめ、過度なリファクタは別タスクに切り出す

### 8-3. 計測との連携

- GTM は本番環境で発火しているので、`/demo` にもそのまま適用される
- Clarity のセッション記録も `/demo` に及ぶ（マスク Strict のままでOK）
- GA4 で `/demo` の `page_view` が取れていれば十分、カスタムイベントは今回不要

### 8-4. llms.txt / JSON-LD の更新

- llms.txt に「デモを提供している」旨の1行を追加（`https://shiftori.app/demo` へのリンクも）
- JSON-LD の `description` にデモがあることを反映
- 虚偽にならないよう、実装済み範囲のみ記載

## 12. 参考：設計検討のログ

本ドキュメントは2026-04-19の壁打ちセッションで決定した内容をまとめたもの。
議論の過程で以下の重要な理解の修正があった。

### 12-1. 「スタッフゼロ負担」の再定義

当初Claudeは「スタッフ側のUI体験を見せること」が重要と誤解していた。
正しくは「店長がスタッフにアプリ登録を促さなくていい」「店長の調整コスト負担がゼロ」が本質であり、スタッフ視点の豪華さは価値の中心ではない。

この理解修正により、「店長視点のみのデモで十分」という設計判断が筋道立って可能になった。

### 12-2. 「すぐ触れる」の哲学化

nataniさんの「他社SaaSの資料請求フローにムカついている」という個人的体験から、シフトリのプロダクト哲学として「誰に対してもゼロ負担」「すぐ触れる・すぐ動かせる」を明文化した。
これはデモだけの話ではなく、シフトリ全体の設計判断の指針として今後機能する。

### 12-3. インタラクティブデモを見送った理由

当初Claudeは「30秒で価値を伝えるインタラクティブデモ」を強く推していた。
