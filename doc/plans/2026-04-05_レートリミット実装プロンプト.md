# レートリミット実装プロンプト

> **前提**: `convex/CLAUDE.md` を必ず先に読むこと
> **参照ファイル**: 末尾の一覧を参照

---

## ゴール

`convex-helpers` の `defineRateLimits` を使い、スタッフ認証系の2つのmutationにレートリミットを追加する。メール爆撃とトークンブルートフォースを防止する。

---

## 背景

- `verifyToken` と `requestReissue` は Clerk 認証なしで誰でも呼べる public mutation
- `requestReissue` はメール送信をトリガーするため、連打されると Resend の従量課金が発生する
- `verifyToken` の UUID v4 トークンは128bitエントロピーがあり現実的な脅威は低いが、保険として制限を入れる
- CLAUDE.md のセキュリティチェックリストに「レートリミットを Magic Link 検証に適用」が未実装として残っている

---

## 実装手順

### 1. スキーマにレートリミット用テーブルを追加

`convex/schema.ts` に `rateLimits` テーブルを追加する。`defineRateLimits` が内部で使用するテーブル。

```ts
// schema.ts に追加
import { defineRateLimits } from "convex-helpers/server/rateLimit";

// テーブル定義は defineRateLimits のドキュメントに従う
// （正確なスキーマ定義は convex-helpers のソースを確認すること）
```

**注意**: `defineRateLimits` が要求するテーブルスキーマの正確な形は、`node_modules/convex-helpers/server/rateLimit.ts` のソースを確認して合わせること。ドキュメントに明示されていない場合がある。

### 2. レートリミット定義ファイルを作成

`convex/_lib/rateLimits.ts` を新規作成：

```ts
import { defineRateLimits } from "convex-helpers/server/rateLimit";

const MINUTE = 60 * 1000;
const HOUR = 60 * MINUTE;

export const { checkRateLimit, rateLimit, resetRateLimit } = defineRateLimits({
  // マジックリンクトークン検証: トークン先頭8文字をキーに
  // 5回/分 — ブルートフォース保険
  verifyToken: {
    kind: "token bucket",
    rate: 5,
    period: MINUTE,
    capacity: 5,
  },

  // リンク再発行リクエスト: email+recruitmentId をキーに
  // 3回/時 — メール爆撃防止（Resend課金対策）
  requestReissue: {
    kind: "token bucket",
    rate: 3,
    period: HOUR,
    capacity: 3,
  },
});
```

### 3. `verifyToken` にレートリミットを適用

`convex/staffAuth/mutations.ts` の `verifyToken` を修正：

```ts
import { rateLimit } from "../_lib/rateLimits";

export const verifyToken = mutation({
  args: { token: v.string() },
  handler: async (ctx, { token }) => {
    // レートリミットチェック（トークン先頭8文字をキーに）
    const key = token.substring(0, 8);
    const { ok, retryAfter } = await rateLimit(ctx, "verifyToken", { key });
    if (!ok) {
      return {
        status: "rate_limited" as const,
        retryAfter,
        recruitmentId: null,
      };
    }

    // 以下は既存ロジックそのまま
    const magicLink = await ctx.db
      .query("magicLinks")
      // ...
  },
});
```

**返り値の型**: 既存の `status: "ok" | "expired"` に `"rate_limited"` を追加。`retryAfter` は秒数（number）。

### 4. `requestReissue` にレートリミットを適用

同ファイルの `requestReissue` を修正：

```ts
export const requestReissue = mutation({
  args: {
    email: v.string(),
    recruitmentId: v.id("recruitments"),
  },
  handler: async (ctx, { email, recruitmentId }) => {
    // レートリミットチェック（email + recruitmentId をキーに）
    const key = `${email}:${recruitmentId}`;
    const { ok } = await rateLimit(ctx, "requestReissue", { key });
    if (!ok) {
      // メアド列挙攻撃防止: レートリミットでも成功時と同じレスポンス（void）を返す
      return;
    }

    // 以下は既存ロジックそのまま
    const staff = await ctx.db
      .query("staffs")
      // ...
  },
});
```

**重要**: `requestReissue` はレートリミットに引っかかっても **void を返す**。エラーにしない。理由：成否を区別するとメアド列挙攻撃の手がかりになるため（既存のセキュリティ方針と同じ）。

### 5. フロントエンド側の対応

`verifyToken` の返り値に `"rate_limited"` が追加されるため、フロントで表示を対応する。

**対象ファイル**: magic link のトークン検証を呼んでいるコンポーネント（ルーティングや認証フロー周辺）を確認し、`status === "rate_limited"` の場合のUI表示を追加する。

- 表示テキスト例: `「しばらく時間を置いてから再度アクセスしてください」`
- `retryAfter` を使って「◯秒後に再試行できます」と表示してもよい

`requestReissue` は返り値が変わらない（void のまま）ので **フロント側の変更不要**。

### 6. CLAUDE.md のチェックリスト更新

`convex/CLAUDE.md` のセキュリティ実装チェックリストを更新：

```diff
- [ ] レートリミットを Magic Link 検証に適用
+ [x] レートリミットを Magic Link 検証に適用
```

### 7. テスト

`convex/staffAuth/mutations.test.ts` にレートリミットのテストを追加：

**verifyToken のレートリミットテスト**:
- 同じキーで6回連続呼び出し → 6回目で `status: "rate_limited"` が返ること
- 異なるキーなら制限されないこと

**requestReissue のレートリミットテスト**:
- 同じ email + recruitmentId で4回連続呼び出し → 4回目でもエラーにならないこと（void が返る）
- メール送信 action がスケジュールされた回数が3回であること（4回目はサイレントに弾かれる）

## 設計判断メモ

| 判断 | 理由 |
|------|------|
| `convex-helpers` の `defineRateLimits` を使用 | プロジェクトに既に `convex-helpers` がインストール済み。`@convex-dev/rate-limiter` コンポーネントは高機能だがMVP段階ではオーバー |
| `requestReissue` でレートリミット時もvoidを返す | メアド列挙攻撃防止の既存方針と一貫させる |
| `verifyToken` のキーをトークン先頭8文字にする | 同一トークンへの連打を防ぎつつ、異なるトークンは独立して検証可能にする |
| フロントの `rate_limited` 表示はシンプルに | UXとしては「少し待ってね」が伝われば十分 |
