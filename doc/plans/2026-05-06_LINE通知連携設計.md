# LINE通知連携設計

**作成日**: 2026-05-06
**関連バックログ**: #336 LINE連携
**ステータス**: 設計確定、実装プロンプト作成待ち
**決定の場**: Claudeとの壁打ちセッション

---

## 1. 背景

### なぜ今LINE連携か

- LINEは日本の店舗内コミュニケーションの事実上の標準であり、メールよりも到達率・既読率が高いと想定される
- 飲食店スタッフ層（特に若年層）はメールをほとんど開かない傾向
- 競合（airShift、らくしふ、oplus等）のLINE対応は限定的または有償
- AIO経由の流入時、機能リストにLINE通知があることは差別化要素になり得る

### 検証段階のスタンス

- 実ユーザーから「LINE通知が欲しい」という声は **未取得**
- Natani自身の仮説段階での実装着手であることを認識した上で進める
- 「最初のユーザーが欲しいと言うまで実装しない」自分ルールから一時的に逸脱
- ただし「ハッピーパス優先・スコープクリープ回避」原則は維持し、機能を最小限に絞る

### ゼロ負担哲学との整合

シフトリのコア価値「スタッフ・店長双方へのゼロ負担」を維持する設計とする。具体的には：

- スタッフはLINE連携を **強制されない**（メール運用のままでも完全に動く）
- 店長は連携を **管理しない**（連携状況を気にする必要がない）
- 通知チャネルの切り替えは **自動フォールバック方式** で実現（後述）

## 2. スコープ

### IN（このフェーズで実装する）

- スタッフのLINE連携フロー（QR/URL → LINEログイン → lineUserId保存）
- シフト確定通知のLINE送信（連携済みスタッフのみ）
- 連携依頼メール（個別 / 一括）
- 既存通知メールへの「LINEに切り替える」ボタン埋め込み
- 自動フォールバック（未連携・友達解除時はメール）
- friend Webhook対応（follow / unfollow で状態同期）
- LINE上のあらゆる受信メッセージへの定型応答（Reply API）
- Quota API監視（上限到達時はメール退避）

### OUT（このフェーズでは実装しない）

- 個別の連絡（店長↔スタッフのチャット代替）
- グループトーク機能
- 対話型ボット（自然言語解釈、シフト確認コマンドなど）
- リッチメニュー
- LIFF（LINE内Webアプリ）
- LINEログイン以外の認証フロー（Messaging APIアカウント連携機能は不採用）
- LINE残量警告UI（warning状態の可視化）

## 3. 設計判断の経緯

### 3-1. 連携フロー：LINEログイン版を採用

3つの選択肢を比較した。

| 方式 | 採用 | 理由 |
|---|---|---|
| LIFF | × | スタッフにLINE内ブラウザでの操作を強いる、開発コスト高 |
| Messaging APIアカウント連携 | × | 「先に友達追加 → 認証」の3ステップ、UX重い |
| **LINEログイン** | **○** | **2タップで完了、シフトリ起点でフロー設計可能** |

LINEログインの`bot_prompt=aggressive`オプションで、認可画面に友達追加チェックボックスを目立たせる。これにより1フローで「lineUserId取得」と「友達追加」を同時完了できる。

### 3-2. 通知チャネル切り替え：自動フォールバック方式を採用

3案を比較した。

| 案 | 設定UI | 採用 |
|---|---|---|
| A. 店舗単位設定 | 店長が一括選択 | × |
| B. ユーザー単位設定 | スタッフが個別選択 | × |
| **C. 自動フォールバック** | **設定UIゼロ** | **○** |

C案は「LINE連携済み・友達追加中のスタッフはLINE、それ以外はメール」という分岐を通知送信時に自動判定する。設定UIが不要なため、店長・スタッフ双方の認知負荷が増えない。ゼロ負担哲学と整合する。

### 3-3. 友達解除時の挙動：Webhook追跡＋自動メール退避

LINEログイン認可だけではPush送信できないため、友達追加状態の追跡が必須。`follow` / `unfollow` Webhookで`lineFollowing`を同期し、友達解除時は自動でメールにフォールバックする。

### 3-4. Quota管理：normal / exceeded の2状態のみ

当初は normal / warning / exceeded の3状態を検討したが、warning状態の可視化はスコープから外した。MVPでは「上限到達時の退避」だけ実装し、UIへの可視化は後回し。

### 3-5. LINE上のメッセージ応答：1パターンのみ

スタッフがLINEに何を送ってきても、Reply APIで「シフトリの通知用アカウントです。シフトの確認や提出は、店長から届くリンクからお願いします」を返す。Reply APIは課金対象外のため、コスト面のリスクはゼロ。対話機能・自然言語解釈はv2以降。

### 3-6. コスト方針

| 段階 | プラン | 月額 | 通数 | トリガー |
|---|---|---|---|---|
| 開始 | コミュニケーション | 0円 | 200通 | デフォルト |
| 拡張 | ライト | 5,000円 | 5,000通 | 200通枠が足りなくなった時点 |
| 検討 | スタンダード | 15,000円 | 30,000通+従量 | 5,000通超え兆候時に有料化を再検討 |

Natani自己負担の上限は **月5,000円**（ライトプランまで）。それ以上の規模に達した時点で「シフトリ有料化」または「LINE通知だけ有料オプション化」を再判断する。

## 4. UX設計

### 4-1. スタッフのLINE連携フロー

```
① 店長がスタッフ管理画面で「LINE連携用QR/URL発行」をクリック
   ↓
② Convexがそのスタッフ専用トークンを生成（72時間有効、1回使用で無効化）
   ↓
③ 店長UIにQRと「○○さんにメールで送る」ボタンが表示される
   ↓
④ スタッフがQRを読む or メールのURLを踏む
   ↓
⑤ LINEログイン認可画面が開く（友達追加チェックボックスON）
   ↓
⑥ スタッフが認可承認
   ↓
⑦ コールバックでaccess_token受け取り → プロフィールAPIでuserId取得
   ↓
⑧ Convexがトークンとマッチして該当ユーザーにlineUserIdを保存
   ↓
⑨ followイベントWebhookでlineFollowing: trueに更新
   ↓
⑩ 完了画面「LINEで通知を受け取れるようになりました」
```

スタッフ視点では **「QR読む → 認可するボタン押す → 完了」の2タップ**。

### 4-2. 連携依頼メール

#### 個別送信

店長が「○○さんに連携依頼を送る」ボタンを押すと、そのスタッフに以下のメールが届く。

```
件名: シフトリのLINE通知が使えるようになります

○○さん

シフトのお知らせをLINEでも受け取れるようになりました

[ LINEに切り替える ]

メールのままでも問題なく届きます
```

最後の一文「メールのままでも問題なく届きます」がゼロ負担哲学の表現。

#### 一括送信

店長UIに「未連携スタッフ全員に連携依頼を送る」ボタンを用意。文面は個別送信と同じテンプレート。導入直後の一斉案内に使う。

### 4-3. 既存通知メールへのボタン埋め込み

すべての通知メール（シフト確定、提出依頼など）の末尾に「LINEに切り替える」ボタンを追加。条件分岐は以下：

| スタッフ状態 | ボタン |
|---|---|
| 未連携 | 「LINEに切り替える」 |
| 連携済み・友達OK | （そもそもLINEで届くため、メール送信されない） |
| 連携済み・友達解除 | 「LINEを再連携する」（注釈付き） |

### 4-4. 通知振り分けロジック

通知送信時、Convex側で以下の判定を行う。

```typescript
function selectChannel(user: User, quotaStatus: QuotaStatus): "line" | "email" {
  if (quotaStatus.status === "exceeded") return "email";
  if (user.lineUserId && user.lineFollowing) return "line";
  return "email";
}
```

### 4-5. Quota上限到達時のUI

LINE上限に達した状態で店長がシフト確定操作を行った場合：

- 操作自体は通常通り完了
- 完了スナックバーに「（今月はメール通知です）」と注釈を追記

例：

```
シフトを確定しました
（今月はメール通知です）
```

### 4-6. LINEへの受信メッセージ応答

スタッフが公式LINEに何かメッセージを送った場合、Reply APIで以下を返す。

```
シフトリの通知用アカウントです 📅
シフトの確認や提出は
店長さんから届くメール / LINEのリンクからお願いします
```

連携状態に関わらず全員に同じメッセージを返す（判定処理を入れない）。Reply APIは課金対象外。

## 5. データ設計

### 5-1. usersテーブル拡張

```typescript
users: defineTable({
  // 既存フィールドは省略

  lineUserId: v.optional(v.string()),       // LINEユーザーID（連携完了時に保存）
  lineLinkedAt: v.optional(v.number()),     // 連携日時
  lineFollowing: v.optional(v.boolean()),   // 友達追加中フラグ（Webhookで同期）
})
  .index("by_line_user_id", ["lineUserId"])
```

### 5-2. lineLinkTokens（新規）

```typescript
lineLinkTokens: defineTable({
  userId: v.id("users"),
  token: v.string(),
  expiresAt: v.number(),       // 発行時刻 + 72時間
  usedAt: v.optional(v.number()),
})
  .index("by_token", ["token"])
  .index("by_user_id", ["userId"])
```

### 5-3. lineQuotaStatus（新規・単一レコード）

```typescript
lineQuotaStatus: defineTable({
  checkedAt: v.number(),
  totalQuota: v.number(),       // 当月の上限通数
  consumed: v.number(),          // 当月の消費通数
  remaining: v.number(),         // 残量
  status: v.union(
    v.literal("normal"),
    v.literal("exceeded"),
  ),
  plan: v.union(
    v.literal("communication"),
    v.literal("light"),
    v.literal("standard"),
  ),
})
```

このテーブルは常に1レコードだけ持つ。cron で更新、通知送信時は読むだけ（LINE APIは叩かない）。

## 6. 技術設計

### 6-1. LINEチャネル構成

LINE Developersコンソールで以下を作成：

- プロバイダー: 1つ（例: `shiftori`）
- LINEログインチャネル: 1つ（認可フロー用）
- Messaging APIチャネル: 1つ（通知送信・Webhook受信用）

両チャネルを同じプロバイダー配下に置くことで、`userId`が共通になる。

### 6-2. Webhook受信エンドポイント

Convexの`httpAction`でWebhookを受ける。

```
POST /line/webhook
```

#### 受信イベントと処理

| イベント | 処理 |
|---|---|
| `follow` | 該当`lineUserId`の`lineFollowing`を`true`に更新 |
| `unfollow` | `lineFollowing`を`false`に更新 |
| `message` | 定型応答メッセージをReply APIで返す |

#### シグネチャ検証

`X-Line-Signature`ヘッダーでHMAC-SHA256検証を実装する。既存のWebhookシグネチャ検証パターン（Resend等で使用中）に倣う。

### 6-3. LINEログインフロー（OAuth 2.0）

```
1. 認可URL生成
   GET https://access.line.me/oauth2/v2.1/authorize
     ?response_type=code
     &client_id={LOGIN_CHANNEL_ID}
     &redirect_uri={CALLBACK_URL}
     &state={LINK_TOKEN}
     &scope=profile openid
     &bot_prompt=aggressive

2. ユーザーが認可 → コールバックにcode到達

3. アクセストークン取得
   POST https://api.line.me/oauth2/v2.1/token

4. プロフィール取得
   GET https://api.line.me/v2/profile
   → userId 取得

5. lineLinkTokens を state で検索 → userId保存
```

### 6-4. Push通知送信

```
POST https://api.line.me/v2/bot/message/push
{
  "to": "{lineUserId}",
  "messages": [
    { "type": "text", "text": "シフト確定のお知らせ ..." }
  ]
}
```

メッセージ形式はテキスト（MVP）。Flex Messageは v1.5 以降で検討。

### 6-5. Quota監視

```
GET https://api.line.me/v2/bot/message/quota
GET https://api.line.me/v2/bot/message/quota/consumption
```

#### 監視タイミング

- Convex cronで1日1回（深夜）
- 通知送信前のguardは入れない（DB値だけ見る）

#### 状態遷移

```
normal: remaining > 0
exceeded: remaining <= 0
```

通知送信時は`lineQuotaStatus`を読んで判定。`status === "exceeded"`なら全員メール送信。

### 6-6. QRコード生成

連携用URLからQRコードを動的生成。`qrcode`パッケージでSVG生成し、店長UIに表示。印刷・スクリーンショット利用に耐えるよう余白付き。

## 7. 段階的実装計画

各ステップは独立リリース可能な単位。順番に実装し、各完了時点で動作確認する。

### Step 1: LINEチャネル開設＋Webhook受信基盤

**目的**: LINE側との通信路を確立する

**完了条件**:
- LINE Developersコンソールでプロバイダー、LINEログインチャネル、Messaging APIチャネルを作成
- Convex `httpAction`で`/line/webhook`エンドポイントを実装
- シグネチャ検証パスのテストが通る
- LINE Developersコンソールの「Webhook検証」ボタンが成功する
- 環境変数（`LINE_LOGIN_CHANNEL_ID`、`LINE_LOGIN_CHANNEL_SECRET`、`LINE_MESSAGING_CHANNEL_ACCESS_TOKEN`、`LINE_MESSAGING_CHANNEL_SECRET`）を `.env` に追加

**注意**:
- LINE側の設定でハマりやすい（特にWebhook URLの検証）
- Convex deployment URLを事前にWebhook URLとして登録する必要あり

### Step 2: Convexスキーマ拡張

**目的**: データ層を整備する

**完了条件**:
- `users`テーブルに`lineUserId`、`lineLinkedAt`、`lineFollowing`を追加
- `lineLinkTokens`テーブルを新規作成
- `lineQuotaStatus`テーブルを新規作成
- マイグレーション（`@convex-dev/migrations`使用）
- `lineUserId`のユニーク性チェック（既に他ユーザーが使用中ならエラー）

### Step 3: LINEログイン認可フロー

**目的**: スタッフがQR/URL経由でLINE連携できる

**完了条件**:
- 店長UIで「LINE連携用QR/URL発行」ボタン → トークン生成 → QR表示
- スタッフが踏むと認可URLにリダイレクト
- 認可後コールバックでアクセストークン → プロフィール取得 → `lineUserId`保存
- `bot_prompt=aggressive`で友達追加チェックON
- E2Eテスト追加（モック使用）

**注意**:
- `state`パラメータに`lineLinkTokens`のtokenを使う（CSRF対策と紐付けの両立）
- トークン使用時に`usedAt`を記録、再使用時はエラー

### Step 4: Webhook処理（follow / unfollow / message）

**目的**: LINE側状態の同期と定型応答

**完了条件**:
- `follow`イベントで`lineFollowing: true`更新
- `unfollow`イベントで`lineFollowing: false`更新
- `message`イベントで定型応答メッセージを返す（Reply API）
- レートリミット適用（`convex-helpers`使用）

### Step 5: 通知振り分けロジック

**目的**: 既存通知をLINE / メールで自動振り分け

**完了条件**:
- シフト確定通知の送信処理に`selectChannel`関数を組み込む
- LINE Push送信実装（`POST /v2/bot/message/push`）
- メールフォールバック動作確認
- 既存メール送信ロジックは変更最小限に留める

### Step 6: 既存メールテンプレート修正

**目的**: 通知メールから連携導線を作る

**完了条件**:
- すべての通知メールテンプレートに「LINEに切り替える」ボタンを追加
- スタッフ状態に応じたボタン文言の出し分け
- ボタンのリンク先は連携用QR/URLと同じトークン経由フロー

### Step 7: 連携依頼メール（個別 + 一括）

**目的**: 店長から能動的に連携を依頼できる

**完了条件**:
- 店長UIにスタッフ別「連携依頼を送る」ボタン
- 店長UIに「未連携スタッフ全員に送る」ボタン
- 連携依頼用メールテンプレート作成
- 各ボタンクリックで対応するスタッフにメール送信

### Step 8: Quota管理

**目的**: 上限到達時の事故を防ぐ

**完了条件**:
- Convex cronで`lineQuotaStatus`を1日1回更新
- 通知送信時に`status === "exceeded"`なら全員メール送信に切替
- スナックバーに「（今月はメール通知です）」を追記
- 月初リセット時の自動復帰

### Step 9: 動作確認とリリース準備

**目的**: 統合テストとドキュメント整備

**完了条件**:
- E2Eテスト一通り
- privacy policy更新（LINE連携部分の追記）
- 利用規約更新（LINE連携時のデータ取扱）
- staging環境での通し検証
- リリースノート作成
