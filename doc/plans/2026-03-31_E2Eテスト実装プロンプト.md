## ゴール

田中さん（居酒屋店長）が初めてYPSを使い、シフトを確定してスタッフにメール通知するまでの一連のフローをE2Eテストとして実装する。

## テストシナリオ概要

**ファイル名**: `e2e/scenarios/first-shift-delivery.test.ts`

**テスト名**: `田中さんの初めてのシフト確定`

9ステップの `test.step()` で構成。POMクラスは `DashboardPage` と `ShiftBoardPage` の2つ。

---

## ステップ詳細

### Step 1: 初回セットアップを完了する

**操作**:
1. `/dashboard` にアクセス
2. SetupModal が自動表示される（`shop === null` 時に強制表示）
3. Step 1（form id: `setup-step1`）に入力:
   - 店舗名: `テスト居酒屋`
   - シフト開始時間: `10:00`
   - シフト終了時間: `23:00`
4. 「次へ」ボタンをクリック
5. Step 2（form id: `setup-step2`）に入力:
   - あなたの名前: `田中太郎`
   - メールアドレス: `tanaka@example.com`
6. 「登録する」ボタンをクリック

**アサーション**:
- トースト「セットアップが完了しました」が表示される
- SetupModal が閉じる
- ダッシュボードのコンテンツが表示される

**注意**: SetupStep1 のシフト時間帯は `<select>` または `<input>` で実装されている可能性あり。実際のUIコンポーネントを確認して適切なセレクターを使うこと。

---

### Step 2: スタッフを追加する

**操作**:
1. 「スタッフを追加」的なボタンをクリック → モーダル表示（タイトル: `スタッフを追加`）
2. AddStaffForm（form id: `add-staff-form`）は初期状態で3行の入力欄がある
3. 2名分を入力（3行目は空のままでOK、空行はバリデーションで無視される）:
   - 1行目: 名前 `鈴木花子` / メール `suzuki@example.com`
   - 2行目: 名前 `佐藤次郎` / メール `sato@example.com`
   - 3行目の削除ボタン（`aria-label="削除"`）をクリックして3行目を消す
4. モーダルの「登録する」ボタンをクリック

**アサーション**:
- トースト「スタッフを追加しました」が表示される
- モーダルが閉じる
- ダッシュボードのスタッフセクションに3名表示される（田中太郎 + 鈴木花子 + 佐藤次郎）

---

### Step 3: シフト募集を作成する

**操作**:
1. 募集作成ボタンをクリック → モーダル表示（タイトル: `新しい募集を作成`）
2. CreateRecruitmentForm（form id: `create-recruitment-form`）に入力:
   - シフト期間開始: 来週月曜の日付（動的に計算）
   - シフト期間終了: 来週日曜の日付（動的に計算）
   - 提出締切日: 今週金曜の日付（動的に計算）
3. 「作成する」ボタンをクリック

**アサーション**:
- トースト「募集を作成しました」が表示される
- モーダルが閉じる
- 募集セクションにカードが表示される（期間テキストが含まれる）

**注意**: 日付入力は `<input type="date">` なので `page.fill()` で `YYYY-MM-DD` 形式の値を入力する。日付は `Date` からヘルパー関数で動的に生成すること（ハードコードしない）。

---

### [テストセットアップ] API でシフトデータを投入

Step 3 の後、Step 4 の前に実行。`test.step()` にはしない（UI操作ではないため）。

**方法**: `page.evaluate()` で Convex client を使い `shiftAssignments` を直接挿入する、または `convex run` CLI でシードスクリプトを実行する。

**投入データ**: 3名 × 7日 = 最大21レコードの `shiftAssignments`
- 田中太郎: 月〜金 10:00-18:00（土日は休み）
- 鈴木花子: 月水金 11:00-19:00（火木土日は休み）
- 佐藤次郎: 火木土 14:00-23:00（月水金日は休み）

**注意**: `recruitmentId` と `staffId` はStep 1-3で作成されたデータから取得する必要がある。方法の候補:
1. `page.evaluate()` で Convex query を叩いて取得
2. Step 3の後でURLからrecruitmentIdを取得（募集カードの「シフトボードを開く」のonClickにrecruitmentIdが含まれる）
3. Convex の `api.dashboard.queries.getDashboardData` をテスト内で呼んで取得

実装者がやりやすい方法で実装して良い。ただし、テストの安定性を最優先すること。

---

### Step 4: シフトボードを開いて全体を確認する

**操作**:
1. 募集カードの「シフトボードを開く」ボタンをクリック
2. `/shiftboard/{recruitmentId}` に画面遷移

**アサーション**:
- URL が `/shiftboard/` を含むこと
- **注意: コードのデフォルトビューは `daily`（日別）**。情報設計書とはずれているが、コードの実際の動作に合わせる
- ガントチャートが表示され、3名のスタッフ名が見えること（田中太郎、鈴木花子、佐藤次郎）
- シフトバーが表示されていること（APIで投入したデータ）

---

### Step 5: 日別ビューでシフトを調整する

**操作**:
1. ガントチャート上で、田中太郎のバーの右端をドラッグして30分短縮する（18:00 → 17:30）

**ドラッグの実装**:
- バーの右端の座標を取得
- `page.mouse.move()` → `page.mouse.down()` → `page.mouse.move()` → `page.mouse.up()` で30分分（=1スロット分）左に移動
- バーを特定するためのセレクター: `data-testid` が付いていない場合はスタッフ名テキストから行を特定し、その行内のバー要素を取得する
- **ドラッグが安定しない場合**: このステップのドラッグ部分をスキップ（コメントアウト）してもよい。その場合、Step 8 の「変更が反映」アサーションも調整する

**アサーション**:
- バーの時間テキストが変化すること（可能であれば）

**注意**: CLAUDE.md の方針: 「ガントチャートのドラッグ操作はE2Eでは最小限の検証のみ（精密な時間検証はユニットテスト側）」。座標計算が複雑で不安定な場合は、ドラッグの代わりに「バーが表示されていること」のみを検証する形に簡素化して構わない。

---

### Step 6: 別の日を確認する

**操作**:
1. 日付タブの2番目（火曜日）をクリック

**アサーション**:
- 火曜日のタブがアクティブ状態になる
- 火曜日にシフトがあるスタッフのバーが表示される（佐藤次郎の14:00-23:00）
- 火曜日にシフトがないスタッフの行にはバーがない（鈴木花子）

---

### Step 7: 保存する

**操作**:
1. ヘッダーの「保存」ボタンをクリック

**アサーション**:
- トースト「保存しました」が表示される

---

### Step 8: 一覧ビューで最終確認する

**操作**:
1. ヘッダーの SegmentGroup で「一覧」をクリック

**アサーション**:
- 一覧ビュー（テーブル形式）が表示される
- 全スタッフ名が表示される
- 日別の時間テキスト（例: `10:00-18:00`）または「休」が表示される

---

### Step 9: シフトを確定して通知する

**操作**:
1. ヘッダーの「スタッフに送信する」ボタンをクリック
2. 確認ダイアログが表示される（タイトル: `スタッフにシフトを送信しますか？`）
3. ダイアログ内に「対象: 3名」と期間が表示されていることを確認
4. 「送信する」ボタンをクリック

**アサーション**:
- トースト「送信しました」が表示される
- 確認ダイアログが閉じる
- ヘッダーに「送信済み」テキストが表示される
- ボタンのラベルが「再送信する」に変わる

**注意**: メール送信の成否は検証しない（CLAUDE.md: 「UI層の成功トースト+送信済み表示のみ検証」）

## テストデータのヘルパー

### 日付ヘルパー（`e2e/helpers/date.ts`）

```ts
import dayjs from 'dayjs';

/** 来週月曜〜日曜の日付を返す */
export function getNextWeekDates() {
  const nextMonday = dayjs().add(1, 'week').startOf('week').add(1, 'day');
  return {
    periodStart: nextMonday.format('YYYY-MM-DD'),
    periodEnd: nextMonday.add(6, 'day').format('YYYY-MM-DD'),
    deadline: dayjs().endOf('week').subtract(1, 'day').format('YYYY-MM-DD'), // 今週金曜
    dates: Array.from({ length: 7 }, (_, i) => nextMonday.add(i, 'day').format('YYYY-MM-DD')),
  };
}
```

## 実装時の注意事項

1. **`page.waitForTimeout()` 禁止**。`expect().toBeVisible()` 等で待機する
2. **CSSセレクター禁止**（`.chakra-button` 等）。`getByRole` / `getByText` > `getByTestId` の優先順位
3. **トーストの検証**: Chakra UI のトーストは `role="status"` で取れる可能性あり。実際のDOM構造を確認すること
4. **SegmentGroup**: Chakra UI v3 の SegmentGroup は `role="radio"` or `role="tab"` で取れる可能性あり。実装を確認
5. **モーダルのsubmit**: モーダル内のフォームは `form id` + 外部の `type="submit"` ボタンで紐づいている。submitは「モーダルフッターのボタンクリック」で行う
6. **date input**: `<input type="date">` への入力は `page.fill(selector, 'YYYY-MM-DD')` で行う
7. **Convex リアルタイム**: Convex はリアルタイムなので、API でデータ投入後にページをリロードしなくてもUIに反映される（ただし `useQuery` のresubscriptionタイミングによっては少し待つ必要がある）
8. **コードのデフォルトビュー**: `ShiftBoardPage` の `useState<ViewMode>("daily")` — デフォルトは日別ビュー。情報設計書のHP-01（一覧がデフォルト）とはずれているが、コードに合わせる
