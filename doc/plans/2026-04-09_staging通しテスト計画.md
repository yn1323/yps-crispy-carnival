## 0. 背景

リリース前の最後のタスク。**全ての他タスクが反映された staging 環境** で、実際に手動で通しテストを行う。
自動テスト（Vitest / Playwright E2E）は CI で通っている前提。ここでは手動でのハッピーパス／エッジケース確認と、見つかったバグを潰す時間を含む。

> ⚠️ このタスクは **#2, #3, #4, #6, #7 の後に実行する**。全てのリリース要素がマージされた状態で staging をいじる。

---

## 1. 準備

### 1-1. staging 環境の用意

- Cloudflare Pages に staging ブランチのプレビュー URL を用意
- 環境変数：
    - `VITE_CLERK_PUBLISHABLE_KEY`（staging インスタンス）
    - `VITE_CONVEX_URL`（staging デプロイメント）
- Convex の staging デプロイメントに最新スキーマがデプロイされていること
- `pnpm convex:env:setup` で staging 環境変数が同期されていること

### 1-2. テストデータの用意

- 管理者アカウント × 1（自分）
- マネージャーアカウント × 1（別メアドで招待）
- スタッフ × 3（別メアド、もしくはエイリアス `user+test1@gmail.com` 等）
- テスト店舗 × 1（「テスト店舗」）
- 過去・現在・未来のシフト募集 × 各1件

### 1-3. 事前チェック

- [ ] `pnpm lint` / `pnpm type-check` / `pnpm test` / `pnpm e2e` CIが通っている
- [ ] Convex の `_generated` が最新
- [ ] 環境変数に漏れ／ずれがない

## 3. バグ発見時の運用

### 3-1. 分類
発見したバグは重要度で分類する。

| レベル | 定義 | 対応 |
|--------|-----|-----|
| **P0** | リリースブロッカー。ハッピーパスが通らない、データ破壊、セキュリティ | 即日修正 |
| **P1** | 体験の劣化。回避策はあるが恥ずかしい | リリース前までに修正 |
| **P2** | マイナー。リリース後で良い | GitHub Issue 化して先送り |

### 3-2. 記録フォーマット

GitHub Issue に以下のテンプレで起票。

```
## 概要
（何が起きたか1行）

## 再現手順
1.
2.

## 期待動作

## 実際の動作

## 環境
- デバイス:
- ブラウザ:
- ユーザー種別:
- URL:

## スクショ／ログ

## 重要度
P0 / P1 / P2
```
