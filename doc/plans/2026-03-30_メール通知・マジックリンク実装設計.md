## 1. 概要

シフト確定後、スタッフにメール通知を送信し、マジックリンク経由で全員の確定シフトを閲覧できるようにする。

### MVPスコープ

| 対象 | MVP |
|------|-----|
| シフト確定メール | ✅ |
| 募集開始メール | ❌ MVP後 |
| マジックリンク再発行ページ | ✅ |

---

## 2. 画面一覧

### 2-1. 確定シフト閲覧ページ（スタッフ向け・新規）

| 項目 | 内容 |
|------|------|
| URL | `/shifts/view?token={token}` |
| 認証 | トークンまたはセッション（後述） |
| 表示内容 | 全スタッフの確定シフト（壁貼りシフト表のデジタル版） |
| 編集 | **不可**（閲覧専用） |
| IDOR対策 | URLにrecruitmentIdやstaffIdを含めない。トークンのみでアクセス制御。トークンはUUID v4で推測不可能 |

### 2-2. マジックリンク再発行ページ（スタッフ向け・新規）

| 項目 | 内容 |
|------|------|
| URL | `/shifts/reissue` |
| 認証 | 不要（公開ページ） |
| 表示内容 | メールアドレス入力フォーム → 「メールを送信しました」画面 |
| IDOR対策 | メアドの存在有無を返さない。登録済み→メール送信、未登録→何もしない。どちらも同じ「メールを送信しました」画面を表示 |
| 送信対象 | 入力されたメアドに紐づくスタッフ本人の最新募集のみ。他人のリンクは発行しない |

### 2-3. 既存画面の変更

| 画面 | 変更内容 |
|------|----------|
| シフトボード（一覧ビュー） | 「シフトを確定する」ボタン押下時にメール送信を実行 |
| シフトボード（一覧ビュー） | 確定済みの場合「再送信する」ボタンを表示。押下で同じメールを再送信（新トークン発行） |

## 3. メール仕様

### 3-1. シフト確定メール

| 項目 | 内容 |
|------|------|
| 送信タイミング | 田中さんが「シフトを確定する」ボタンを押した時 |
| 送信先 | 募集対象の全スタッフ（個別送信。各自に自分のシフト情報が入る） |
| 送信元 | Resend |
| 差出人名 | 店舗名（例：居酒屋さくら） |
| 差出人アドレス | `noreply@yps.app` |
| 件名 | `【{店舗名}】{開始日}〜{終了日} シフト確定のお知らせ` |
| 例 | `【居酒屋さくら】1/20(月)〜1/26(日) シフト確定のお知らせ` |

### 3-2. メール本文構成

```
{スタッフ名}さん

{開始日}〜{終了日} のシフトが確定しました。

┌─────────────────────────────┐
│ あなたのシフト                │
│                             │
│ 1/20 (月)  10:00 - 18:00   │
│ 1/21 (火)  休み              │
│ 1/22 (水)  12:00 - 21:00   │
│ 1/23 (木)  休み              │
│ 1/24 (金)  10:00 - 18:00   │
│ 1/25 (土)  休み              │
│ 1/26 (日)  休み              │
└─────────────────────────────┘

        [ 全員のシフトを確認する ]
          ↑ マジックリンク

このリンクは24時間有効です。
リンクを開くと14日間閲覧できます。
期限切れの場合はこちらから再発行 → {再発行ページURL}

─────────────────────────────
シフトについてのご質問・変更希望は
お店に直接ご連絡ください。
※ このメールに返信しても届きません。
```

### 3-3. メール本文の仕様

| 項目 | 内容 |
|------|------|
| シフト表示 | 全日表示（休みの日も「休み」と明示） |
| 出勤日の書式 | `{日付} ({曜日})  {開始時刻} - {終了時刻}` |
| 休みの書式 | `{日付} ({曜日})  休み` |
| リンクボタン | 「全員のシフトを確認する」→ マジックリンクURL |
| 再発行リンク | 「期限切れの場合はこちらから再発行」→ 再発行ページURL |
| フッター | 返信不可の注意 + お店に連絡の案内 |

### 3-4. 再送信時

| 項目 | 内容 |
|------|------|
| トリガー | 田中さんが「再送信する」ボタンを押下 |
| 動作 | 新しいトークンを発行し、同じ形式のメールを全スタッフに再送信 |
| 旧トークン | 旧トークンも有効期限内であれば引き続き有効（無効化しない） |

## 4. 認証・セキュリティ仕様

### 4-1. トークン（マジックリンク）

| 項目 | 値 | 理由 |
|------|-----|------|
| 生成方式 | 自前実装（Clerk不使用） | スタッフにClerkアカウントを作らせない「負担ゼロ」原則 |
| トークン形式 | UUID v4（暗号学的に推測不可能） | — |
| 有効期限 | **24時間** | 労働時間から収入が推定できるため、リンク漏洩リスクを限定する |
| 使用回数 | 複数回可（有効期限内） | 24時間と短いため、ワンタイムにする必要性が低い |
| 紐づけ | 募集ID（全スタッフ共通のシフト表を表示） |
| 送信方法 | Resend経由のメール |

### 4-2. セッション

| 項目 | 値 | 理由 |
|------|-----|------|
| 発行タイミング | スタッフがトークン付きURLを初回クリックした時 |
| 保存方法 | **localStorage** | Convex（WebSocket通信）ではHttpOnly Cookieからの値取得が不可。SPAアーキテクチャに適合 |
| 有効期限 | **14日間固定** | シフト期間＋余裕。24時間トークンの短さを補完 |
| セッション切れ時 | 再発行ページへ誘導 |

### 4-3. 認証フロー

```
【通常フロー】
1. 田中さんが「シフトを確定する」→ Resendでメール送信（トークン付きURL）
2. スタッフがリンクをクリック（24時間以内）
3. Convexでトークン検証 → 有効 → セッションID発行
4. フロントがlocalStorageにセッションID保存
5. 確定シフト閲覧ページを表示
6. 以降14日間はセッションIDでアクセス可能

【トークン期限切れ・セッション切れ】
1. スタッフがリンクをクリック or ページにアクセス
2. トークン期限切れ＆セッションなし → 再発行ページへリダイレクト
3. メアド入力 → Convexでメアド照合 → 該当あればResendで新リンク送信
4. 画面は一律「メールを送信しました」表示（メアド存在有無を漏らさない）
```

### 4-4. IDOR対策まとめ

| 画面 | 対策 |
|------|------|
| 確定シフト閲覧ページ | URLにトークンのみ。recruitmentId/staffIdを含めない |
| 再発行ページ | メアド入力→一律「送信しました」。存在有無を返さない |
| 再発行で送るリンク | 入力メアドに紐づく本人の最新募集のみ |

## 5. 実装タスク

### Convex（バックエンド）

| # | タスク | 種類 | 概要 |
|---|--------|------|------|
| 1 | トークン生成 | mutation | 募集IDに紐づくUUID v4トークンを生成。有効期限24時間 |
| 2 | トークン検証 | query | トークンの有効期限チェック。有効ならシフトデータ返却 |
| 3 | セッション発行 | mutation | トークン検証成功時にセッションID（UUID v4）を生成。有効期限14日 |
| 4 | セッション検証 | query | セッションIDの有効期限チェック。有効ならシフトデータ返却 |
| 5 | メール送信（確定） | action | Resend APIで確定メールを全スタッフに個別送信 |
| 6 | メール再送信 | action | 新トークン発行 → Resend APIでメール再送信 |
| 7 | リンク再発行 | action | メアド照合 → 該当あれば新トークン生成 → Resend送信 |

### フロントエンド

| # | タスク | 概要 |
|---|--------|------|
| 8 | 確定シフト閲覧ページ | 新ルート `/shifts/view`。トークンまたはセッションで認証し、全員のシフト表を表示（閲覧専用） |
| 9 | 再発行ページ | 新ルート `/shifts/reissue`。メアド入力 → 送信完了の2状態 |
| 10 | シフトボード：確定ボタン改修 | 「シフトを確定する」押下時にメール送信actionを呼び出す |
| 11 | シフトボード：再送信ボタン | 確定済みの場合に「再送信する」ボタンを表示 |
| 12 | セッション管理 | localStorage にセッションID保存。アクセス時にセッション検証queryを呼び出す |
| 13 | リダイレクト処理 | トークン期限切れ＆セッションなし → 再発行ページへリダイレクト |

### メールテンプレート

| # | タスク | 概要 |
|---|--------|------|
| 14 | 確定メールテンプレート | Resend用HTMLメールテンプレート。スタッフ名・シフト一覧・リンク・注意書きを含む |

## 7. セキュリティ・認証設計書（3/28）からの変更点

| 項目 | 旧（3/28） | 新（本書） | 変更理由 |
|------|-----------|-----------|----------|
| トークン有効期限 | 72時間 | **24時間** | 労働時間→収入推定リスクを限定 |
| トークン使用回数 | ワンタイム | **複数回可** | 24時間と短いためワンタイム不要 |
| セッション有効期限 | 7日間 | **14日間固定** | シフト確認の利便性を優先 |
| セッション保存方法 | Secure Cookie（HttpOnly） | **localStorage** | Convex WebSocket通信でHttpOnly Cookieの値をJS側で取得不可。SPA構造に適合 |
| 認証基盤 | Clerk前提 | **自前実装** | スタッフにClerkアカウント不要（「負担ゼロ」原則） |
| メール送信 | 未定 | **Resend** | — |

> **注意**: セキュリティ・認証設計書の該当箇所もこの内容で上書き更新すること。
