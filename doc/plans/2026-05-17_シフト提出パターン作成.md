# 提出方法ごとの時間入力整理 実装計画

## Summary

スタッフ希望の提出方法ごとに、必要な時間入力だけを出す形へ変更する。

- `時間指定`: シフト開始/終了時間を入力する。
- `日毎指定`: 時間は入力しない。出勤可能日だけ保存する。
- `勤務区分`: 区分名と区分ごとの開始/終了時間を入力する。
- シフト表の最終的な見せ方は後回し。今回の実装では日毎指定のデータ保存・提出画面・既存フロー非破壊までを対象にする。

DB全削除前提なので、既存データ互換や migration/backfill は作らない。

## Key Changes

- `shops` / `recruitments` のグローバルな `shiftStartTime` / `shiftEndTime` 前提をやめ、提出方法の中に必要な時間情報を持たせる。
  - `time`: `{ kind: "time", startTime, endTime }`
  - `dateOnly`: `{ kind: "dateOnly" }`
  - `shiftType`: `{ kind: "shiftType", options: [{ id, name, startTime, endTime, sortOrder }] }`
- `createRecruitment` は店舗の `submissionPattern` をそのまま募集にスナップショットする。
- `submitShiftRequests` は提出方法ごとに保存先を分ける。
  - `time` / `shiftType`: 既存の `shiftSubmissionSlots` に時間スロットとして保存。
  - `dateOnly`: 新規 `shiftSubmissionDates` テーブルに `date` だけ保存。
- `getSubmissionPageData` は `existingSelection` と前回入力データを提出方法別に返す。
  - `dateOnly` の前回入力は「曜日だけ」のパターンとして返す。
  - `shiftType` は前回スロットの時間帯が現在の区分時間と一致する場合だけ `optionId` に変換する。

## UI Changes

- 初回セットアップは「提出方法を先に選ぶ」構成にする。
  - デフォルト選択は `日毎指定`。
  - `時間指定` を選んだ時だけ開始/終了時間を表示。
  - `勤務区分` を選んだ時だけ区分エディタを表示。
- 店舗設定モーダルも同じ条件分岐にする。
  - グローバルなシフト時間帯セクションは廃止。
  - 勤務区分の時間は区分行の中だけで編集。
- 募集作成確認画面は提出方法の要約だけ表示する。
- スタッフ提出画面は以下に分ける。
  - `time`: 現行の時間入力カード。
  - `dateOnly`: 出勤できる/休みだけのカード。
  - `shiftType`: 休み + 勤務区分チップ。
- シフト表は今回深追いしない。
  - `dateOnly` 募集でも画面が壊れない最低限の分岐を入れる。
  - 時間バーとしての表示や自動配置は次フェーズで設計する。

## Test Plan

- Logic UT
  - setup/edit shop schema: `time` は時間必須、`dateOnly` は時間不要、`shiftType` は区分時間必須。
  - 前回入力変換: `dateOnly` は曜日だけ、`shiftType` は時間完全一致だけ適用。
- Convex Function Test
  - `updateShopSettings`: 各提出方法の保存、勤務区分の不正時刻/重複/空名拒否。
  - `createRecruitment`: 提出方法スナップショット。
  - `submitShiftRequests`: `dateOnly` が日付だけ保存され、時間スロットを作らない。
  - `getSubmissionPageData`: 既存提出と前回入力を提出方法別に返す。
- Scenario Test
  - 初回セットアップから募集作成、日毎指定の提出、提出済み集計まで通る。
  - 店舗設定変更後も既存募集の提出方法スナップショットが変わらない。
- UI/Storybook
  - Setup/EditShopForm の3提出方法 variants。
  - StaffSubmit の `time` / `dateOnly` / `shiftType` stories。
  - Behavior Test を追加する場合は操作検証目的なので `chromatic.disableSnapshot` を付ける。

## Assumptions

- DB全削除前提なので migration は作らない。
- 初回セットアップのデフォルト提出方法は `日毎指定` にする。
- 日毎指定のシフト表での本格表示・配置体験は次フェーズ。今回は保存形式と提出体験を正しく分離し、既存画面が落ちない最低限に留める。
- `0:00-0:00` のような sentinel は使わない。時間がない状態は DB 型で明示する。
