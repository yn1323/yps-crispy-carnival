## 1. 背景

### SEO流入の現状

- shiftori.app のSEO流入が体感ゼロ（Search Consoleで未確認だが要計測）
- AIO対応（llms.txt、JSON-LD）は整備済みだが、SEO流入とは別物

### 既存のSEO実装（資産）

事前調査の結果、SEO基盤は想像以上に整っていた。**新規実装はほぼ不要**。

| 項目 | 現状 |
|---|---|
| 静的HTML生成 | `scripts/prerender.ts` で `/`, `/privacy`, `/terms` をPlaywright経由でビルド時に静的化 |
| メタタグヘルパー | `src/helpers/seo/index.ts` に `buildMeta` / `jsonLdMeta` 完備、noindex対応済み |
| TanStack Router head機能 | 各ルートで `head: () => ({ meta: [...] })` を使用 |
| 構造化データ | TOPに `SoftwareApplication`, `Organization`, `WebSite`, `FAQPage` が既設置 |
| sitemap.xml | `/`, `/privacy`, `/terms` が登録済み |
| robots.txt | 認証必須ページを Disallow 済み |
| デモサンドボックス | `/demo/shiftboard` で実装完了済み（`_unregistered/demo.shiftboard.tsx`）。react-joyrideによるツアーガイド付き |

### 改善すべき問題

1. **TOPのtitleが「シフトリ」だけ** — `buildMeta` の仕様で `title === SITE_NAME` のとき suffix を付けない動作になっており、検索結果のtitleが指名検索しか拾えない最弱の状態
2. **TOPのdescriptionに具体的なペインキーワードが不足** — 現状は「リンクを送るだけで希望シフトを収集」など機能寄り、店長が検索する語と乖離
3. **`/privacy`, `/terms` がインデックス対象になっている** — Google検索結果の `shiftori.app › privacy` がTOPと同等の枠で表示され、サイト全体の評価を希薄化させる可能性
4. **`/demo/shiftboard` に `noindex: true` が指定されている** — シフトリ最大の差別化資産（ゼロ負担で即触れるデモ）が、Googleからは存在しないことになっている。「シフト管理 デモ」「シフト管理 体験版」「シフト管理 試す」のようなコンバージョン直結の購買意図キーワードを完全に取りこぼしている
5. **`/demo/shiftboard` がprerender対象外、sitemap未登録、canonical未設定** — SEO上は完全に透明
6. **`/demo/shiftboard` のSP表示が `SP Coming Soon...` のみ** — モバイルファーストインデクスで低品質判定されるリスクがあり、noindexを外す前に最小限のコンテンツ改修が必要

### タスク2: /privacy, /terms にnoindex設定

#### 対象ファイル

- `src/routes/privacy.tsx`
- `src/routes/terms.tsx`
- `public/sitemap.xml`（タスク3-4でまとめて対応）

#### 変更内容

`buildMeta` の引数に `noindex: true` を追加する。`buildMeta` は既にnoindexオプションを実装済み（`<meta name="robots" content="noindex, nofollow">` を出力する）。

```ts
// privacy.tsx
head: () => ({
  meta: buildMeta({
    title: "プライバシーポリシー",
    description: "シフトリのプライバシーポリシー",
    canonical: "/privacy",
    noindex: true, // ← 追加
  }),
}),

// terms.tsx も同様
```

#### noindexの理由

- 法的ページはSEO評価対象として不要
- 現状、TOPと類似のメタタグでGoogleにインデックスされており、サイト全体の評価を希薄化している可能性
- `nofollow` も同時に付与（`buildMeta` の仕様）。法的ページからの内部リンクは各種ナビゲーションリンク程度で、SEO上の損失は無視できる

#### prerender対象は維持

`scripts/prerender.ts` の `ROUTES` リストに `/privacy`, `/terms` は残す。noindex指定のためにはGoogleがそのページを読みに行ける必要があり、HTMLは引き続き静的生成する。robots.txtでブロックすると逆効果（Googleがnoindexタグを読めなくなる）。

#### robots.txt の確認

`public/robots.txt` で `/privacy` `/terms` をDisallowしていないことを確認する（現状Disallowされていないが念のため）。

#### 3-3. prerender対象に追加

```ts
// scripts/prerender.ts

// Before
const ROUTES = ["/", "/privacy", "/terms"] as const;

// After
const ROUTES = ["/", "/privacy", "/terms", "/demo/shiftboard"] as const;
```

##### 注意：prerender実行時のツアー自動起動

`DemoShiftBoardPage` は `react-joyride` のツアーが自動起動する設計。prerender実行時（headless Chromium）にツアーが起動した状態のHTMLが焼き込まれると、初回訪問者にツアーが「2回目」のように見える可能性がある。

`scripts/prerender.ts` 内では既に以下のフラグを注入している：

```ts
await context.addInitScript(() => {
  (window as unknown as { __PRERENDER__: boolean }).__PRERENDER__ = true;
});
```

**実装時に確認すること**: `DemoIntroTour` および `DemoLauncherFab` コンポーネントが `__PRERENDER__` フラグを参照してツアー起動を抑制しているか。していなければ抑制ロジックを追加する。例：

```tsx
const isPrerendering = typeof window !== "undefined" && (window as unknown as { __PRERENDER__?: boolean }).__PRERENDER__;
if (isPrerendering) return null;
```

##### 注意：prerenderのassertチェック（重要）

`scripts/prerender.ts` 内に以下のチェックがある（失敗するとビルド失敗）：

- HTML最小サイズ 2000バイト
- `<h1>` の存在
- `<title>` タグが2つ以上（index.htmlの静的titleとTanStack Routerのhead出力の両方）
- インラインstyleが5000バイト以上

`/demo/shiftboard` のPC版（`DemoShiftBoardPage`）は明示的な `<h1>` がない可能性が高い。実装時に確認し、なければ追加する：

- 案1: PC版に視覚的な `<h1>` を追加（「シフトリのデモ」など、デザイン的に邪魔にならない位置）
- 案2: PC版に視覚的に隠した `<h1>` を追加（`sr-only` 相当のCSS）
- 案3: SP表示のh1がprerender時に出力されればOK（ただしheadless Chromiumのデフォルトviewportは1280x720でPC版が出る可能性が高いため、案1か2推奨）

prerender実行のviewportを変更する選択肢もあるが、本タスクでは案1または2で対応する。実装時に `pnpm prerender` を実行してassertが通ることを確認すること。

## 5. 注意事項

### LPコピー原則とSEOの両立

シフトリのLPコピー原則は「句読点なし・改行でリズム・短く・削る方向」。これは**HTML本文（h1, h2, p）に対する原則**であるが、本タスクの以下の場所では原則を緩める：

- `<title>`: 検索結果の青リンク文字列。短く・キーワードを前に置く実務SEOの定石を優先
- `<meta name="description">`: 検索結果のスニペット。1〜2文の自然な日本語（句読点あり）が情報伝達効率に勝る
- `/demo/shiftboard` のSP表示テキスト: SEOクロール時に十分な本文として機能させるため、句読点ありの自然な文章で書く

LPの本文・見出し（`src/components/features/LandingPage/`）には触らない。

### 既存の構造化データには触らない

`SoftwareApplication`, `Organization`, `WebSite`, `FAQPage` は既に整備済み。今回は変更しない。

### canonical URL の表記揺れに注意

`buildMeta` 内で `og:url` を `${SITE_URL}${canonical}` で組み立てている。`canonical` は `/` で始まる相対パスを渡す（既存ルートと同じ仕様）。`/demo/shiftboard` も同様。

### prerender の assertRenderedHtml チェック

`scripts/prerender.ts` 内のチェック（HTML最小サイズ・h1の存在・titleタグ2つ以上・インラインstyle最小サイズ）が全て通る必要がある。`/demo/shiftboard` で失敗する可能性があるため、実装後に必ず `pnpm prerender` を実行して通ることを確認すること。

### TanStack Routerの routeTree.gen.ts は自動生成

`src/routes/` 配下のファイルを編集したら、`pnpm dev` 起動中に `src/routeTree.gen.ts` が自動更新される。手動編集禁止（CLAUDE.md規約）。

### ドキュメント更新の判断

`doc/features/` の既存ドキュメントを更新する必要はない。本タスクはSEO設定の最適化であり、機能仕様の変更ではないため。

ただし、本実装計画ドキュメント（`doc/plans/2026-04-25_TOPとデモのSEO最適化_実装計画.md`）を `doc/INDEX.md` のplans一覧に追加することを推奨。

### keyword検証の優先順位

実装着手前に、以下のキーワードを最低限ラッコキーワード等で確認する：

最優先（TOPに反映）:
- 「シフト 集める メール」
- 「メール一通 シフト」
- 「少人数 シフト管理」

最優先（demo/shiftboardに反映）:
- 「シフト管理 デモ」
- 「シフト管理 試す」
- 「シフト管理 登録なし」

検索ボリュームがゼロまたは極小の場合は、コードコメントで残した候補案へ切り替える、または別キーワードを再検討する。
