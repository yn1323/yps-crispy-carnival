# DB設計：ダッシュボード・シフトボード・店舗初回セットアップ

## Context

シフト管理SaaS（YPS v3）のConvexバックエンドDB設計。現在 `convex/schema.ts` は空。フロントエンドのUI/型定義は先行実装済みで、それらと整合するスキーマが必要。

ダッシュボード（店舗管理・スタッフ管理・募集管理）、シフトボード（希望提出・シフト割当・確定通知）、および初回セットアップフロー（店舗情報→募集作成のステッパー）を全てカバーする。

## 命名判断：`shops` を採用

ユーザーの仕様では「store」だが、既存コードベースで `shop` が広く使われている：
- Jotai: `selectedShopAtom`, `hasSelectedShopAtom`
- API modules: `shop/mutations`, `shop/queries`（`_generated/api.d.ts`で確定済み）
- CLAUDE.md: `api.shop.queries.getById`
- features: `src/components/features/Dashboard/` 内で Shop 参照

**フロントエンドとの一貫性を優先し、テーブル名は `shops` とする。** UIラベルのみ「店舗」表示。

## テーブル設計

### 1. `shops`（店舗情報）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `name` | `v.string()` | 店舗名 |
| `shiftStartTime` | `v.string()` | シフト開始時間（例: "14:00"） |
| `shiftEndTime` | `v.string()` | シフト終了時間（例: "25:00" = 翌1:00） |
| `ownerId` | `v.string()` | Clerk userId（作成した管理者） |
| `isDeleted` | `v.boolean()` | 論理削除フラグ |

**インデックス:**
- `by_ownerId`: `["ownerId"]` — 管理者のshop一覧取得

**備考:**
- MVPは1管理者1店舗想定。将来の複数店舗対応のため `ownerId` で紐付け
- 「シフト時間帯」ラベル。注意書き：「営業時間ではなく、仕込み・片付けを含めたスタッフの勤務可能な時間帯を入力してください」

### 2. `users`（管理者ユーザー）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `clerkId` | `v.string()` | Clerk認証ID |
| `name` | `v.string()` | ユーザー名 |
| `email` | `v.string()` | メールアドレス |
| `role` | `v.union(v.literal("admin"), v.literal("manager"))` | 役割 |
| `isDeleted` | `v.boolean()` | 論理削除フラグ |

**インデックス:**
- `by_clerkId`: `["clerkId"]` — Clerk認証後のユーザー検索

**備考:**

---

### 3. `staffs`（スタッフ）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `shopId` | `v.id("shops")` | 所属店舗 |
| `name` | `v.string()` | スタッフ名 |
| `email` | `v.string()` | メールアドレス |
| `isDeleted` | `v.boolean()` | 論理削除フラグ |

**インデックス:**
- `by_shopId`: `["shopId"]` — 店舗のスタッフ一覧
- `by_email`: `["email"]` — メアド検索（マジックリンク再発行時）

**備考:**
- フロントの `Staff` 型には `role: "admin" | "staff"` があるが、DB上スタッフはClerユーザーではない。roleはフロント側で付与（管理者=Clerkユーザー、スタッフ=staffsテーブル）
- 削除時にセッション無効化が必要（セキュリティ設計書）

---

### 4. `recruitments`（シフト募集）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `shopId` | `v.id("shops")` | 対象店舗 |
| `periodStart` | `v.string()` | シフト期間開始日（"2026-01-20"） |
| `periodEnd` | `v.string()` | シフト期間終了日（"2026-01-26"） |
| `deadline` | `v.string()` | 提出締切日（"2026-01-17"） |
| `status` | `v.union(v.literal("open"), v.literal("confirmed"))` | ステータス |
| `confirmedAt` | `v.optional(v.number())` | 確定日時（Unix ms） |
| `isDeleted` | `v.boolean()` | 論理削除フラグ |

**インデックス:**
- `by_shopId`: `["shopId"]` — 店舗の募集一覧
- `by_shopId_status`: `["shopId", "status"]` — ステータス別フィルタ

**備考:**
- フロントの `status: "open" | "completed"` → DB では `"open" | "confirmed"` に変更。「completed」は曖昧（何が完了？）、「confirmed」は情報設計書の「確定」と一致
- `responseCount` / `totalStaffCount` はクエリ時に集計（非正規化しない）
- 確定後も編集可能。再確定で `confirmedAt` 更新＋再通知

---

### 5. `shiftRequests`（スタッフの希望シフト）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `recruitmentId` | `v.id("recruitments")` | 対象募集 |
| `staffId` | `v.id("staffs")` | 提出スタッフ |
| `date` | `v.string()` | 日付（"2026-01-20"） |
| `startTime` | `v.string()` | 希望開始時間（"10:00"） |
| `endTime` | `v.string()` | 希望終了時間（"18:00"） |

**インデックス:**
- `by_recruitmentId`: `["recruitmentId"]` — 募集の全希望取得
- `by_recruitmentId_staffId`: `["recruitmentId", "staffId"]` — スタッフ別希望取得
- `by_staffId`: `["staffId"]` — スタッフ削除時のカスケード用

**備考:**
- 1スタッフ×1日＝1レコード（日ごとに独立）
- 「休み希望」＝該当日のレコードなし
- フロントの `requestedTime: { start, end } | null` と対応。null = レコード不在

---

### 6. `shiftAssignments`（確定シフト割当）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `recruitmentId` | `v.id("recruitments")` | 対象募集 |
| `staffId` | `v.id("staffs")` | 割当スタッフ |
| `date` | `v.string()` | 日付（"2026-01-20"） |
| `startTime` | `v.string()` | 開始時間（"10:00"） |
| `endTime` | `v.string()` | 終了時間（"18:00"） |
| `positionId` | `v.optional(v.id("positions"))` | ポジション（Phase 5） |

**インデックス:**
- `by_recruitmentId`: `["recruitmentId"]` — 募集の全割当取得
- `by_recruitmentId_staffId`: `["recruitmentId", "staffId"]` — スタッフ別割当取得
- `by_recruitmentId_date`: `["recruitmentId", "date"]` — 日別の割当取得

**備考:**
- MVPではポジション機能はPhase 5だが、フロントの `PositionSegment` 型が存在するため `positionId` をoptionalで用意
- 1スタッフ×1日に**複数レコード**可能（ポジション切替で時間帯が分かれる）
- MVPでは1スタッフ×1日＝1レコード（ポジション未使用時）
- フロントの `ShiftData.positions[]` に対応

### 8. `magicLinks`（マジックリンク認証）— MVP

| フィールド | 型 | 説明 |
|---|---|---|
| `token` | `v.string()` | UUID v4トークン |
| `staffId` | `v.id("staffs")` | 対象スタッフ |
| `shopId` | `v.id("shops")` | 対象店舗 |
| `expiresAt` | `v.number()` | 有効期限（Unix ms、72時間後） |
| `usedAt` | `v.optional(v.number())` | 使用日時（ワンタイム制御） |

**インデックス:**
- `by_token`: `["token"]` — トークンでの認証検証
- `by_staffId`: `["staffId"]` — スタッフ削除時のカスケード用

**備考:**
- セキュリティ設計書に基づく: 72時間有効、ワンタイム
- `usedAt` が null → 未使用、値あり → 使用済み
- スタッフ削除時に関連リンクも無効化

## フロントエンド型との対応表

| フロント型 | DBテーブル | 変換ロジック |
|---|---|---|
| `Recruitment` | `recruitments` | `responseCount` = shiftRequestsの提出済みスタッフ数を集計 |
| `Recruitment.totalStaffCount` | `staffs` | shopのアクティブスタッフ数を集計 |
| `Staff` | `staffs` + `users` | roleはクエリ側で付与 |
| `ShiftData` | `shiftAssignments` + `shiftRequests` + `staffs` | JOINしてフロント型に整形 |
| `ShiftData.requestedTime` | `shiftRequests` | 該当日のレコード有無でnull判定 |
| `ShiftData.positions` | `shiftAssignments` + `positions` | MVPでは1レコード=1セグメント |
| `PositionType` | `positions` | そのまま |
| `ConfirmStatus` | `recruitments.confirmedAt` | null = 未確定 |

## 初回セットアップフローとの対応

```
ログイン → ダッシュボード（Empty State）
  ↓
Step 1: 店舗情報登録
  → shops テーブルに INSERT（name, shiftStartTime, shiftEndTime, ownerId）
  ↓
Step 2: シフト募集作成
  → recruitments テーブルに INSERT（shopId, periodStart, periodEnd, deadline, status="open"）
  → staffs の全スタッフに magicLinks 生成 + メール送信
  ↓
ダッシュボード（募集カード表示）
```
