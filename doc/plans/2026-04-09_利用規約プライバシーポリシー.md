## 0. 背景

個人情報（メールアドレス、店舗名、スタッフ名）を扱うサービスなので、**プライバシーポリシーは必須**。
利用規約も免責・禁止事項をカバーする意味で最低限必要。

初回リリースは身内向けなので、**完璧な法務レビュー済みのドキュメントを目指さず、「最低限誠実な内容」を書いて公開する** 方針にする。リリース後に弁護士レビュー等で改善していく。

> ⚠️ 私は弁護士ではないので、最終的には本物のひな形を参考に自分でも確認すること。
> ここで書くのはあくまで「初回リリース向けの叩き台」。

## 1. 方針：コード内で静的ページとして実装

### 選択肢

| 案 | 長所 | 短所 |
|----|-----|-----|
| A. Notion 公開ページ | 更新が楽 | 外部依存・独自ドメインに統合しにくい・OGP ゆらぎ |
| B. アプリ内ルート（`/terms/*`） | ドメイン一貫・UX で一体感 | 更新のたびにデプロイ必要 |
| C. GitHub Pages / 別リポ | コストゼロ | 管理が散らかる |

**選択：B（アプリ内ルート）**。
理由：
- LP のフッターから遷移した先で別ドメインに飛ぶのは UX として不自然
- React Router（TanStack Router）でファイルベースの static route を増やすだけなので工数は半日で収まる
- マークダウンを React 側でレンダリングする形にすれば、更新の心理的ハードルも下がる

### URL
- `/terms/privacy`
- `/terms/terms`

### レイアウト

- 本文：マークダウン相当（`<h1>` + `<h2>` + `<p>`）
- フッター：最終更新日、問い合わせ先

## 3. コンテンツの中身（叩き台）

### 3-1. プライバシーポリシー（骨格）

```
1. 事業者情報
2. 取得する個人情報
   - メールアドレス（管理者・マネージャー・スタッフ）
   - 氏名（店舗名・スタッフ名）
   - 提出されたシフト希望データ
   - Clerk から取得するアカウント情報
3. 利用目的
   - サービスの提供・運用
   - 不具合対応・問い合わせ対応
   - サービス改善のための統計分析
4. 第三者提供
   - 原則行わない
   - 利用している外部サービス（Clerk, Convex, Cloudflare）
5. 外部サービスへの保管
   - Clerk（米国）
   - Convex（米国）
   - Resend（米国）※メール通知に使用する場合
6. Cookie / ローカルストレージの利用
7. セキュリティ対策
8. 開示・訂正・削除の請求方法
9. お問い合わせ先（目安箱の URL）
10. 改定について
11. 制定日・最終更新日
```

### 3-2. 利用規約（骨格）

```
1. 定義
2. 本規約への同意
3. アカウント登録
4. 禁止事項
   - 他者の権利侵害
   - サービスの妨害
   - 虚偽のスタッフ情報登録
   - リバースエンジニアリング
5. 免責事項
   - 現状有姿で提供
   - データ消失の免責（ベストエフォート）
6. サービスの変更・中止
7. 規約の変更
8. 準拠法・管轄裁判所（日本・〇〇地方裁判所）
9. 最終更新日
```

## 4. 実装ステップ

### Step 1: ドラフト作成（2時間）
1. プライバシーポリシーのドラフトを `doc/legal/privacy.md` に書く
2. 利用規約のドラフトを `doc/legal/terms.md` に書く
3. 参考ひな形：
    - [Privacy Policy Generator](https://www.privacypolicies.com/) などの無料ジェネレーターを叩き台に
    - ただし必ず自分でレビューする（コピペ厳禁）

### Step 2: ルート追加（1時間）
1. `src/routes/_unregistered/terms.privacy.tsx` 作成
2. `src/routes/_unregistered/terms.terms.tsx` 作成
3. 各ルートはそのままマークダウンをインポートして表示
4. マークダウン描画：`react-markdown` を導入するか、単純に JSX で書き下す
    - 初回は JSX 直書きでOK。将来的に増えるなら `react-markdown` を検討

### Step 3: 共通レイアウト作成（1時間）
1. `src/components/features/TermsPage/` を作成
    - `index.tsx`：共通レイアウト（ヘッダー・本文・フッター）
    - `constants.ts`：見出し等の共通文言
2. 利用規約／プライバシポリシーの各ページがこのレイアウトを使う
3. Story は1枚でOK（Variants Story で Privacy / Terms の2バリアント）

### Step 4: LP からのリンク接続（30分）
1. `src/components/features/LandingPage/constants.ts` の `LANDING_LINKS` を以下に差し替え
    ```ts
    export const LANDING_LINKS = {
      privacy: "/terms/privacy",
      terms: "/terms/terms",
      feedback: "https://forms.gle/xxxxxxxxx",
    } as const;
    ```

### Step 5: 確認（30分）
1. `pnpm dev` で `/terms/privacy`, `/terms/terms` にアクセスできること
2. ログインしていない状態でも表示されること
3. LP のフッターからリンクが機能すること
4. SP で崩れないこと
5. SubAgent で `pnpm lint`, `pnpm type-check`, `pnpm test` 実行

## 5. ドラフト時の注意

- **事業者情報**：個人開発の場合「個人名＋連絡先」で良い。法人化していない現状は「運営者: natani」＋問い合わせ先（Google Form）の形でOK
- **管轄裁判所**：居住地の地裁を指定
- **14歳未満の扱い**：シフトリのユースケース上、スタッフに未成年がいる可能性がある。保護者の同意が必要な旨を書く
- **海外サーバーの明示**：Convex / Clerk は米国リージョンなので、国外移転の旨を明記（個人情報保護法 27条対応）
- **Cookie / localStorage**：Jotai の `selectedShopAtom` は localStorage 永続化している。これも記載対象
- **問い合わせ先**：個人メアドは晒したくないので **目安箱の Google Form を問い合わせ窓口にする**

## 6. ありがちな落とし穴

### 6-1. コピペひな形そのまま
ネット上のひな形には「当社」等の法人想定文言が残ったまま。個人事業主向けにすべて書き換える。

### 6-2. 最終更新日のタイポ
リリース日を1箇所（`constants.ts`）で管理して、複数箇所に散らさない。

```ts
export const LEGAL_META = {
  effectiveDate: "2026-04-09",
  lastUpdated: "2026-04-09",
} as const;
```

### 6-3. 目安箱の URL が埋まっていない
#7（Google Form 目安箱）の URL が確定してから #3 のリンクを埋める。
未確定ならビルドエラーになるような型にして、漏れを防ぐ。

### 6-4. SEO の `noindex` を付け忘れる
利用規約・プライバシーポリシーは検索結果に出ても問題ないが、身内リリース段階では LP 全体を `noindex` にしておく方が安全。

```html
<meta name="robots" content="noindex, nofollow" />
```

これは OGP 設定（#6）のタスクで対応する。
