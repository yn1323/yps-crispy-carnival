# 店舗情報と初回セットアップ設計

## 日付
2025-03-28

## 背景
シフト編集にあたり、店舗情報（店舗名・シフト時間帯）の登録が必要。
初回ユーザー体験を最優先に設計する。

## 決定事項

### 1. 店舗情報として持つもの
| フィールド | 型 | 説明 |
|---|---|---|
| name | string | 店舗名 |
| shiftStartTime | string | シフト開始時間（例: "14:00"） |
| shiftEndTime | string | シフト終了時間（例: "25:00" = 翌1:00） |

- Convexに `store` テーブルとして作成
- 「営業時間」ではなく「シフト時間帯」というラベリングにする
- 注意書き表示：「営業時間ではなく、仕込み・片付けを含めたスタッフの勤務可能な時間帯を入力してください」

### 2. 初回セットアップフロー
- **形式**: ステッパー式モーダル（1つのモーダル内で2ステップ）
- **導線**: ダッシュボード Empty State → ステッパーモーダル

#### ステップ構成
- **Step 1**: 店舗情報登録（店舗名 + シフト時間帯）
- **Step 2**: シフト募集作成（シフト期間 + 提出締切）
- **完了後**: ダッシュボードに募集カードが表示される

#### フロー全体
```
ログイン → ダッシュボード（Empty State）→ ステッパーモーダル Step1（店舗情報）→ Step2（募集作成）→ ダッシュボード（募集カード表示）
```

### 3. スコープ判断（MVP）

#### 採用
- 店舗名の入力
- シフト時間帯の入力
- 初回ステッパーモーダル（初回専用UI）

#### 不採用・先送り
- 店舗情報の変更機能（MVPでは変更不可）
- Clerk Organization機能（スタッフのマジックリンク認証と設計思想が合わない）
- 住所・電話番号・席数などの詳細店舗情報

### 4. 見送り理由メモ

#### Clerk Organization
- スタッフ全員にClerkアカウントが必要になり「ゼロ負担」のコンセプトが崩れる
- 複数店舗管理や副店長の管理権限が必要になった段階で再検討

#### 営業時間 vs シフト時間帯
- 飲食店では営業時間 ≠ スタッフの勤務時間帯（仕込み・片付けがある）
- 「営業時間」とラベリングすると、仕込み時間が入力できないと誤解される
- 「シフト時間帯」と表現し、注意書きで補足する
