# スタッフ参加QRと承認導線

## 背景

店長がスタッフの名前とメールアドレスを集め、1人ずつ入力する負担を減らす。スタッフ追加の既定導線を「店舗専用QR/URLを共有する」に寄せ、スタッフ本人が名前・メールアドレス・利用規約/プライバシーポリシー同意を入力する。

## 仕様

### 店長側

- ダッシュボードの「スタッフを追加」押下時は、既定で店舗専用QR/URLを表示する。
- 同じモーダル内に「自分で入力する」ボタンを置き、既存の名前・メール入力フォームへ切り替えられるようにする。
- QR経由の申請はすぐ正式スタッフにせず、参加申請として保存する。
- ダッシュボードの「今やること」では、既存のシフトカードを主役として維持し、その下に参加申請がある時だけミニリストを表示する。
- 参加申請ミニリストには名前、メールアドレス、承認ボタン、却下ボタンを表示する。
- 却下は確認ダイアログを挟み、承認は即時処理してToastを表示する。

### スタッフ側

- 店舗専用QR/URLからスタッフ登録ページを開く。
- 画面には「シフトリとは？」の案内、名前、メールアドレス、利用規約/プライバシーポリシー同意を表示する。
- メールアドレスの到達確認、確認コード、メール2回入力はv1では入れない。
- 誤入力対策として、形式チェック、よくあるドメインtypo警告、送信前の確認画面を入れる。
- 確認画面では、このメールアドレスにシフト案内が届くことを明示し、メールアドレスを大きく表示する。
- 申請送信後は「店長の承認をお待ちください」と表示する。

### 法務同意

- QR登録ページで利用規約/プライバシーポリシー同意を取る。
- 申請中はスタッフIDがまだないため、同意バージョン情報は申請テーブルに保持する。
- 店長承認時に正式スタッフを作成し、申請に保持していた同意情報を `legalConsentStates` / `legalConsentEvents` に `method: "staff_registration"` として反映する。
- QR登録から承認されたスタッフには、スタッフ追加時の法務同意メールを送らない。
- 店長が「自分で入力する」から追加したスタッフには、従来通り法務同意メールを送る。

### 通知

- 承認時にLINE連携メールを送る。
- LINE連携メールは既存テンプレートを拡張し、QR申請承認後の文脈では「承認されました。続けてLINE連携できます」という文言にする。
- 承認時に、現在募集中のシフト提出リンクも従来仕様通り送る。
- 手入力追加は従来通り、法務同意メール、LINE連携メール、募集中シフト通知を送る。

### チュートリアル

- 「シフトリへようこそ」4/4には「チュートリアルを閉じる」ボタンを用意する。
- 閉じた状態はDBに保存し、同じ管理者なら別端末でも再表示されないようにする。
- スタッフ数だけで4/4を自動終了させる挙動はやめる。

## DB / API 方針

既存ユーザー・本番データはリリース前に全消しする前提のため、この機能追加ではデータ移行・バックフィル用のmigrationは作成しない。追加テーブルと追加フィールドは新スキーマとしてそのまま扱う。

### 追加テーブル

- `shopRegistrationLinks`
  - `shopId`
  - `token`
  - `createdAt`
  - `revokedAt?`
  - index: `by_token`, `by_shopId`
- `staffRegistrationRequests`
  - `shopId`
  - `name`
  - `email`
  - `emailNormalized`
  - `status`: `pending` / `approved` / `rejected`
  - `termsConsentVersion`
  - `privacyConsentVersion`
  - `termsDocumentVersion`
  - `privacyDocumentVersion`
  - `consentedAt`
  - `approvedStaffId?`
  - `reviewedAt?`
  - `reviewedByUserId?`
  - `createdAt`
  - index: `by_shopId_status`, `by_token` は持たない

### 追加/変更API

- `staffRegistration.queries.getRegistrationPageData`
  - tokenから店舗名と法務文書情報を取得する。
- `staffRegistration.mutations.submitRegistrationRequest`
  - token、名前、メール、同意フラグを受け取り、参加申請を作成する。
- `staffRegistration.queries.getPendingRequests`
  - 店長向けに自店舗のpending申請を返す。
- `staffRegistration.mutations.approveRequest`
  - 参加申請を承認し、正式スタッフ作成、同意コピー、通知予約を行う。
- `staffRegistration.mutations.rejectRequest`
  - 参加申請を却下する。
- `staffRegistration.mutations.ensureShopRegistrationLink`
  - 店長向けに店舗固定登録リンクを作成/取得する。
- `dashboard.mutations.dismissOnboarding`
  - 管理者のチュートリアル終了状態をDB保存する。

## テスト方針

- Logic UT
  - メールtypo警告。
  - スタッフ登録フォームの確認画面へ進める条件。
- UI Component Test / Storybook
  - スタッフ追加モーダルのQR表示、URLコピー、手入力切替。
  - スタッフ登録ページの入力、確認、申請完了、無効リンク、typo警告、同意未チェック。
  - ダッシュボードの参加申請0件、1件、複数件、承認/却下操作。
  - 「シフトリへようこそ」4/4の終了ボタン。
- Convex Function Test
  - 登録リンク作成/取得、申請作成、重複メール、無効リンク、削除済み店舗。
  - 店長だけがpending申請を読めること。
  - 承認で `staffs` 作成、同意状態コピー、LINE連携メール予約、募集中シフト通知予約。
  - 却下でスタッフ作成・通知予約をしないこと。
  - 手入力追加は従来通り法務同意メールを予約すること。
- Convex Scenario Test
  - 店舗QR表示、スタッフ申請、店長承認、スタッフ一覧反映、同意済み扱い、LINE連携メール/募集中シフト通知予約までの一連の流れ。
  - 申請却下、既存スタッフ重複、削除済み店舗の代表ケース。
