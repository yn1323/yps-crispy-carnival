## 1. 設計方針

### 大原則：セキュリティはデータの機密度に比例させる

過剰防衛でプロダクトの価値を殺さない。YPSの最大の差別化は「スタッフ負担ゼロ（アプリもアカウントも不要）」であり、セキュリティ設計はこの価値を維持したまま個人情報を適切に保護する。

### YPSが扱うデータの機密度

| データ | 機密度 | 備考 |
|--------|--------|------|
| スタッフの名前 | 低 | 職場内で公知の情報 |
| スタッフのメールアドレス | 中 | 個人情報だが単体では限定的 |
| シフト時間帯（名前との組み合わせ） | 中 | 勤務実態がわかり給与推定が可能。個人情報として扱う |
| 給与額・口座情報・マイナンバー | — | **YPSでは一切扱わない** |

### 競合との比較

| サービス | 扱うデータ | スタッフ認証 |
|----------|-----------|-------------|
| マネーフォワードクラウド給与 | 給与額・口座・マイナンバー（最高機密） | パスワード設定＋2段階認証 |
| Airシフト | シフト＋勤怠（中〜高） | 専用アプリ＋AirIDログイン |
| **YPS** | **シフト時間帯のみ（中）** | **マジックリンク＋セッション** |
| 現状（LINE + Excel） | 全員分のシフト表が平文で共有（無防備） | なし |

YPSへの移行だけで、現状（LINE + Excel）より確実にセキュリティは向上する。

## 2. 管理者（田中さん）の認証

| 項目 | 内容 |
|------|------|
| 認証方式 | **Clerk**（既存のID/パスワード認証） |
| 対象 | 店長（管理者は1名想定） |
| 備考 | 管理者はスタッフ全員分のシフトや個人情報にアクセスできるため、通常のログイン認証を使用 |

## 3. スタッフの認証

### 3-1. 認証フロー

```
【初回アクセス】
1. 田中さんが募集を作成 → スタッフにメール自動送信
2. メールにマジックリンク（ワンタイムURL）が含まれる
3. スタッフがリンクをクリック → セッション発行
4. リンクは即座に無効化される（ワンタイム）
5. 以降はセッション（Cookie）でアクセス

【セッション切れ後の再アクセス】
1. スタッフがYPSにアクセス → セッション切れ画面
2. メールアドレスを入力
3. 新しいマジックリンクがメールで届く
4. リンクをクリック → 新セッション発行
```

### 3-2. マジックリンク仕様

| 項目 | 値 | 理由 |
|------|-----|------|
| 有効期限 | **72時間** | 田中さんが水曜に送信 → スタッフが木〜金に開くケースをカバー |
| 使用回数 | **ワンタイム**（1回クリックで無効化） | リンク漏洩時のリスクを最小化 |
| トークン | UUID v4（暗号学的に推測不可能） | — |
| 配信方法 | メールのみ | LINE連携はPhase 3 |

### 3-3. セッション仕様

| 項目 | 値 | 理由 |
|------|-----|------|
| 有効期限 | **7日間** | シフト1週間分をカバー。30日だと漏洩時のリスク窓が広すぎる |
| 保存方法 | Secure Cookie（HttpOnly, SameSite=Strict） | XSS/CSRF対策 |
| 通信 | HTTPS必須 | 通信経路の暗号化 |
| セッション切れ時 | メアド入力 → 新マジックリンク再送信 | セルフサービスで田中さんの手間ゼロ |

### 3-4. 閲覧範囲

| 項目 | 内容 |
|------|------|
| 閲覧可能データ | **全スタッフのシフト**（自分含む） |
| 理由 | 飲食店では「誰と一緒のシフトか」を確認する必要がある |
| セキュリティ上の影響 | 1つのセッション漏洩で全員分のシフト情報が閲覧される可能性がある。ただしセッション有効期限7日＋Secure Cookieで窓を限定 |

## 4. 管理者による無効化

| 操作 | 効果 |
|------|------|
| スタッフを削除 | そのスタッフのセッションを即座に無効化 |
| マジックリンクの再発行 | 古いリンク（未使用の場合）を無効化し、新しいリンクを送信 |

## 5. 個人情報保護の方針

### 5-1. 法的な位置づけ

- シフト表（名前＋勤務時間帯）は個人情報に該当する
- 職場内での共有は黙示的同意があると見なされるが、外部への漏洩は禁止
- YPSはインターネット経由でアクセスさせるため、適切なアクセス制御が必須

### 5-2. YPSの対策

| リスク | 対策 |
|--------|------|
| URLの推測 | UUID v4トークンにより推測は事実上不可能 |
| リンクの転送・共有 | ワンタイムリンクのため、転送されても使用済みならアクセス不可 |
| メールの流出 | リンクはワンタイムなので、流出しても使用後は無効。セッションはブラウザ紐付け |
| スマホの紛失・盗難 | セッション有効期限7日で窓を限定。管理者から無効化も可能 |
| 不正な第三者のアクセス | Secure Cookie + HTTPS で保護 |

### 5-3. YPSが持たない情報（意図的に持たない）

- 生年月日
- 住所
- 電話番号
- 給与額・時給
- 口座情報
- マイナンバー

個人情報の保有量を最小限にすることで、漏洩時のリスクを最小化する。

## 6. MVPスコープの判断

| 項目 | MVP | 理由 |
|------|-----|------|
| マジックリンク認証 | ✅ | コア機能 |
| セッション管理（7日） | ✅ | UXに直結 |
| HTTPS + Secure Cookie | ✅ | 基本的なセキュリティ |
| ワンタイムリンク | ✅ | セキュリティの基盤 |
| メアド入力による再発行 | ✅ | セルフサービス |
| 管理者によるセッション無効化 | ✅ | スタッフ退職時に必要 |
| 自動保存 | ❌ MVP外 | 手動保存ボタンで十分 |
| パスワード認証 | ❌ 不要 | プロダクト価値を損なう |
| 2段階認証 | ❌ 不要 | データ機密度に対して過剰 |

## 7. 調査に基づく根拠

本設計は以下の調査・分析に基づく：

- **NN/g（Nielsen Norman Group）**: ユーザーは長年の経験から「明示的に保存を指示しないとデータが消える」と学習している。保存ボタンの維持が推奨される
- **ui-patterns.com**: 自動保存を実装しても保存ボタンを消すべきではない。ユーザーはボタンがないとパニックになる
- **個人情報保護法**: シフト表は社内共有は黙示的同意、外部漏洩は禁止。適切なアクセス制御が必要
- **マジックリンクのベストプラクティス**: ワンタイム＋短期有効期限が標準。セッションと組み合わせることでUXとセキュリティを両立
- **競合分析**: Airシフトはアプリ+ログイン必須、給与明細SaaSはパスワード+2段階認証。YPSはデータ機密度（中）に見合ったマジックリンク方式で差別化
