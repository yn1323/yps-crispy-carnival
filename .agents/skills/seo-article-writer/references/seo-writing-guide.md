# 日本語SEO記事ガイド

この参照は、シフトリのArticleSiteに日本語SEO記事を書くための判断基準です。
2026-06-23時点で、日本語版Google検索セントラル、国内SEO実務記事、文化庁の日本語文書方針、アクセシビリティ文書を確認して整理しています。

## 基本方針

SEO記事は、検索順位を操作する文章ではなく、検索して来た人の用事を終わらせる文章として書きます。

シフトリの記事では、次を最優先します。

- 小さなお店の店長、管理者、シフト作成担当者が読める。
- 読者が「次に何をすればいいか」まで分かる。
- シフト作成、希望回収、未提出確認、共有、調整の実務に戻れる。
- 文章だけでなく、表、箇条書き、画像で判断を助ける。
- 商品紹介だけの記事にしない。

Google公式は、有用で信頼できるユーザー第一のコンテンツを重視しています。
国内SEO実務記事でも、検索意図を読み取り、ユーザーに有益な内容にすること、タイトル、内容、alt、書式、E-E-A-Tを整えることが共通して語られています。

## 検索意図の決め方

1本の記事で満たす検索意図は1つに絞ります。

主な型：

- 方法：`LINEでシフト希望を集める方法`
- 比較：`紙・LINE・Excel・アプリの違い`
- チェックリスト：`シフト表を作る前に確認すること`
- 例文：`希望シフト提出をお願いするメッセージ例`
- 失敗対策：`シフト希望が集まらないときの見直し`
- 製品導線：`シフトリで希望回収から共有まで進める流れ`

検索意図を決めるときは、次を分けて考えます。

- 読者は誰か。
- 何に困って検索しているか。
- すぐ知りたい答えは何か。
- 読後にどの作業へ進むか。
- どの情報があれば検索を終えられるか。

広いテーマは分割します。
たとえば「シフト管理 アプリ」は比較記事向きで、「LINE シフト 希望 集め方」は手順記事向きです。

## キーワードの扱い

キーワードは、検索者の言葉を記事に反映するために使います。
詰め込みのためには使いません。

手順：

1. ユーザーの依頼文から主要語を抜く。
2. 店長や管理者が検索しそうな自然な表現に直す。
3. 関連語を2〜5個だけ選ぶ。
4. タイトル、description、冒頭、H2のどこに自然に入るかを見る。
5. 不自然なら入れない。

避けること：

- 同じ語句を何度も繰り返す。
- 似た検索語ごとに薄い別記事を作る。
- `メタキーワード` 的な発想で単語を並べる。
- 読者が使わない専門語を見出しにする。

## タイトル

タイトルは、検索結果と記事一覧で読む人に向けた約束です。

基準：

- 何の記事か一目で分かる。
- 主要キーワードを前半に自然に入れる。
- クリックを煽らない。
- 既存記事と似すぎない。
- 目安は日本語で26〜38字程度。短くても意味が通ればよい。

良い例：

- `LINEでシフト希望を集める方法`
- `希望シフトの提出期限を決めるコツ`
- `Excelでシフト表を作る前に整理すること`

避ける例：

- `完全ガイド`
- `徹底解説`
- `最新版`
- `知らないと損する`
- `シフト管理を効率化する方法`

強い言葉は、本当にその範囲を扱うときだけ使います。

## description

descriptionは、記事カードと検索結果で読者の判断を助ける説明です。

基準：

- 80〜120字を目安にする。
- 1〜2文で書く。
- タイトルをそのまま繰り返さない。
- 読者の悩みと読後の成果を入れる。
- 最後は句点で終える。

例：

```md
LINEでシフト希望を集めるときに決めておきたい提出期限、書き方、未提出確認の流れを整理します。小さなお店で始めやすい運用例も紹介します。
```

## 構成

基本は結論先出しです。

1. 冒頭：誰の何の悩みに答えるかを書く。
2. 冒頭2段落以内：結論か最初に決めることを書く。
3. 前半：手順、判断基準、チェックリストを置く。
4. 中盤：例、比較、よくある失敗を置く。
5. 後半：シフトリで楽になる部分を必要な範囲で説明する。
6. 最後：短く整理し、自然なCTAを置く。

標準の見出し例：

```md
## まず決めること

## LINEで希望を集める手順

## 未提出を見落とさない確認方法

## 紙やExcelと併用するときの注意点

## シフトリでまとめると楽になること

## まとめ
```

H2が3つ以上あるとArticleSiteの目次が出ます。
目次で読んでも内容が分かる見出しにします。

## 見出し

見出しは、読者が拾い読みするための道しるべです。

基準：

- `注意点`、`ポイント`、`まとめ` だけにしない。
- その節で答える問いが分かる形にする。
- できれば読者の作業順に並べる。
- H3は、H2が長くなりすぎるときだけ使う。

弱い例：

```md
## 注意点
```

良い例：

```md
## 提出期限はシフト作成日から逆算して決める
```

弱い例：

```md
## 便利な方法
```

良い例：

```md
## LINEだけで集めると未提出を見落としやすい
```

## 文体

シフトリの記事は、店長や管理者に向けた実務メモのように書きます。

基準：

- `です/ます` を基本にする。
- 説教しない。
- スタッフを責めない。
- 大げさなマーケティング表現を使わない。
- 「重要です」で終わらせず、何をすればよいかを書く。
- 専門語を出すときは、先に普通の言葉で説明する。

避ける言い回し：

- `重要なのは`
- `効率化できます`
- `しっかり管理しましょう`
- `多角的に`
- `包括的に`
- `課題を深掘りします`
- `最適化しましょう`
- `言えるでしょう`

使いやすい言い回し：

- `まず、提出期限を決めます。`
- `次に、出してほしい日付範囲を伝えます。`
- `紙で集める場合は、転記する人を決めておくと漏れを減らせます。`
- `LINEで集める場合は、提出済みかどうかを別の表で確認できる状態にします。`

## 文の長さ

日本語SEO記事では、短い文を基本にします。

目安：

- 1文は40〜70字程度。
- 90字を超えたら分割を検討する。
- 1文に入れる主張は1つにする。
- 因果関係が長くなるときは、結論と理由を分ける。

悪い例：

```md
LINEでシフト希望を集める場合、スタッフから個別にメッセージが届くため確認しやすいように見えますが、締切後に誰が出していないかを追うには別の管理表が必要になり、結果として確認漏れが起きやすくなります。
```

良い例：

```md
LINEでシフト希望を集めると、スタッフから個別にメッセージが届きます。
一見すると確認しやすそうですが、締切後に「誰が出していないか」を追うには別の管理表が必要です。
その表がないと、確認漏れが起きやすくなります。
```

## 読みやすい改行

改行は、読みやすさのために使います。
見た目を細切れにするためには使いません。

ArticleSiteのMarkdownでは、段落ごとに空行を入れます。
1文ごとに必ず空行を入れると、画面が間延びして読みにくくなります。

基準：

- 1段落は1つの話題にする。
- 1段落は1〜3文を基本にする。
- 4文以上続く段落は分割する。
- スマホで6行以上になりそうな段落は分割する。
- 手順、条件、比較、確認項目は箇条書きにする。
- 箇条書きは5項目程度までにし、長くなる場合は小見出しを分ける。
- 1文だけの段落を連打しない。強調したい結論だけに使う。
- 改行で余韻を作らない。読者が読み飛ばしやすい単位を作る。

段落の分け方：

```md
シフト希望を集める前に、提出期限と提出方法を決めます。
この2つが曖昧なままだと、スタッフごとに連絡方法が分かれて確認が難しくなります。

LINEで集める場合は、提出先を1つのトークに決めます。
紙や口頭で受けた希望も、あとで同じ表に転記できるようにしておくと確認漏れを減らせます。
```

箇条書きにした方がよい例：

```md
シフト希望の案内には、次の3点を入れます。

- 提出期限
- 希望を出してほしい日付範囲
- 変更したい場合の連絡方法
```

改行で避けること：

- 意味のない1文段落の連続。
- 1項目が2〜3行になる長い箇条書き。
- 見出し直下に空っぽの前置きを置く。
- PCでは読めるがスマホで塊に見える長段落。

## 導入文

導入文は短くします。
背景を長く語らず、検索者の悩みと記事の答えをすぐ出します。

良い導入：

```md
LINEでシフト希望を集めると、スタッフは出しやすくなります。
一方で、提出済みの確認や締切後の集計を決めておかないと、店長側の確認作業が増えます。

この記事では、LINEで希望を集める前に決めること、実際の案内文、未提出を見落とさない確認方法を整理します。
```

避ける導入：

```md
シフト管理は店舗運営において非常に重要な業務です。
近年、多くの店舗で効率化が求められており、さまざまな方法が注目されています。
```

## 本文に入れる実用要素

標準記事では、次のうち2つ以上を入れます。

- 手順
- チェックリスト
- 比較表
- 例文
- よくある失敗
- 判断基準
- 運用の例外
- シフトリで置き換えられる作業

例文は短く、実際に送れる形にします。

```md
来月分のシフト希望を集めます。
6月25日（水）までに、入れる日と休みたい日をこのLINEに送ってください。
変更がある場合も、同じLINEに返信してください。
```

## 表と箇条書き

比較は表にします。
手順は番号付きリストにします。
条件や確認項目は箇条書きにします。

表の列は3〜4列までにします。
スマホで読みにくい表は、箇条書きに分けます。

表の例：

```md
| 方法 | 向いている店 | 注意点 |
|---|---|---|
| LINE | スタッフが普段からLINEを見る店 | 未提出確認を別で管理する |
| 紙 | 店内で全員に会える店 | 転記漏れに注意する |
| Excel | 作成担当者が表計算に慣れている店 | スマホ提出とは相性が悪い |
```

## 内部リンク

内部リンクは、読者の次の作業につながる場所に置きます。

基準：

- 標準記事は2〜5本程度。
- 最初に自然に出た箇所で1回だけリンクする。
- アンカーテキストは具体的にする。
- `こちら`、`詳しくはこちら` だけにしない。
- `relatedSlugs` は隣の悩みに進める記事を選ぶ。

良い例：

```md
[勤務区分ごとに希望を集める方法](/articles/shift-type-request-guide)も合わせて確認すると、早番・遅番がある店舗でも整理しやすくなります。
```

## シフトリの出し方

シフトリは、記事の主役にしすぎません。
読者の困りごとを説明し、手作業での対策を示したあとに、自然な選択肢として出します。

順序：

1. 店舗運用の困りごとを書く。
2. 手作業での対策を書く。
3. その対策で残る手間を書く。
4. シフトリで置き換えられる作業を書く。
5. デモや登録への導線を短く置く。

禁止：

- 冒頭から商品紹介にする。
- 未確認の機能を断定する。
- 時短効果や導入実績を根拠なく書く。
- 競合や既存運用を雑に下げる。

公開ページのブランド表記は変わることがあります。
記事内でブランド名や公開訴求を書くときは、現在の公開ページやリポジトリの表示を確認します。

## E-E-A-Tと信頼性

シフトリ記事での信頼性は、派手な肩書きよりも「確認できることを正確に書く」ことです。

基準：

- `author: "シフトリ編集部"` を基本にする。
- 実績、数値、料金、機能、外部制度は確認してから書く。
- 労務、法務、給与、個人情報に関わる内容は、必ず公的・公式情報を確認する。
- 不確かなことは断定しない。
- 古い記事の日付だけを新しくしない。
- `updatedAt` は内容を実質的に変えたときだけ更新する。

## 画像

画像は、本文の理解を助けるときだけ入れます。

入れるとよい画像：

- 希望回収から確認までの流れ。
- 紙、LINE、Excel、アプリの比較。
- 未提出チェックのイメージ。
- カレンダーやシフト表の整理例。
- 店長とスタッフのやり取り。

入れない方がよい画像：

- ただの雰囲気画像。
- 本文と同じことを薄く繰り返す画像。
- 何を示しているか分からない抽象図。
- 文字が多すぎる画像。

頻度：

- 短い記事：0〜1枚。
- 標準記事：1〜2枚。
- 長い比較・手順記事：最大4枚。

配置：

- 画像で説明する本文の直後か直前に置く。
- 画像の近くに、何を見るべきかが分かる文章を置く。
- キャプションは1文で要点を書く。

alt：

- 画像の見た目だけでなく、記事内での役割を書く。
- キーワードを詰め込まない。
- 装飾画像なら空altを検討する。ただしArticleSiteの記事画像は基本的に説明目的にする。

良いalt：

```md
LINEで集めた希望を、未提出チェックとシフト表に分けて整理する流れ
```

弱いalt：

```md
スマホとカレンダーのイラスト
```

## 生成AIを使うとき

生成AIは、構成案、例文案、画像案、表のたたき台に使えます。
ただし、そのまま公開しません。

必ず確認すること：

- 機能や料金を作っていないか。
- 読者の悩みに対して具体的な答えになっているか。
- 既存記事と重複していないか。
- 日本語がシフトリの温度感に合っているか。
- 空句や一般論だけで段落が成立していないか。
- 画像の文字が崩れていないか。

## よくないSEO記事の兆候

- 冒頭が長い一般論になっている。
- 見出しだけ見ると何の記事か分からない。
- `重要です` と言うだけで、作業が書かれていない。
- キーワードが不自然に繰り返されている。
- 既存記事とほぼ同じ内容になっている。
- 画像が本文と関係ない。
- シフトリの紹介が早すぎる。
- 検索者が読み終えても次の作業に進めない。

## 執筆前チェック

- 検索語は自然な日本語か。
- 読者は具体的に想像できるか。
- 記事は1つの検索意図に答えているか。
- 既存記事と重複していないか。
- H2を読むだけで流れが分かるか。
- 画像を入れる理由があるか。
- CTAは読者の次の行動として自然か。

## 公開前チェック

- 冒頭2段落で答えが出ているか。
- 1段落が長すぎないか。
- スマホで読んだときに文章の塊が重くないか。
- 箇条書きや表で整理すべき箇所が段落のまま残っていないか。
- titleとdescriptionが本文と一致しているか。
- 画像、alt、キャプション、近くの本文が同じ内容を指しているか。
- 未確認の断定がないか。
- ArticleSiteのテストが通っているか。
