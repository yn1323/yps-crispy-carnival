# UI Architect

UIに触る前に必ずこのワークフローを通す。
コードを書き始める前に、最低でも「ゴール分解」「コンポーネント選定」「状態と文言の方針」を短く言語化してユーザーに共有する。
コピーだけの依頼でもこのスキルを使う。

> この職能の本質: ユーザーのタスクを翻訳して、人間が迷わず達成できる形に落とす。
> カッコよさより、使いやすさ・伝わりやすさ・速さが最上位。
> 見た目は、業務の意味と状態が伝わったあとで整える。

---

## 必ずやること

1. `doc/rules/testing-strategy.md` を読む。
2. Convexコードを扱う場合は `convex/_generated/ai/guidelines.md` も読む。
3. UI実装なら、対象の既存コンポーネント・Story・近い実装を確認する。
4. UI文言なら、画面の利用者、要素、プロダクト上の事実を確認する。
5. ユーザーからUI指摘・好み・「前に言った」系の発言を受けたら、このスキルを開いて、再利用できる内容か判定する。

このリポジトリでは Vite / Storybook / Convex dev server はユーザーが起動する。
新規で起動しない。
表示確認が必要なときは、既に起動している画面・Storybook・スクリーンショット・コード上のStoryを使う。

## ワークフロー

### 0. 証拠を集める

- 対象画面、近い既存UI、Storybook story、関連ドキュメントを確認する。
- 「現状のUI」「前の指摘」「ユーザーの好み」が関係する場合は、メモリーを短く検索する。
- UI文言が通知、削除、権限、課金、法務同意、バックグラウンド処理を説明する場合は、コードで実際の挙動を確認する。
- ブラウザ表示の指摘を受けた場合は、可能なら実画面・スクリーンショット・Storybookで見える状態を優先する。

### 1. ゴール分解

ユーザーの要求を以下に分ける。

- プライマリータスク: 1画面1つ。ここに来て一番やりたい行動。
- セカンダリータスク: 補助・副次的な行動。
- 捨てるタスク: この画面ではやらせないこと。
- ユーザーコンテキスト: 管理者かスタッフか、初回か日常操作か、PCかスマホか、急いでいるか不安か。
- データ形状: 1件か多件か、読み中心か編集中心か、静的か頻繁更新か。
- プロダクトの事実: すぐ完了するのか、受け付けただけか、後で通知されるのか、誰に見えるのか。

プライマリーが2つ以上ある画面は分解失敗。
画面分割するか、1つを主、残りを従に格下げする。
詳細は `references/decomposition.md`。

### 2. 情報アーキテクチャ

データ形状と利用頻度からページ骨格を決める。

| シナリオ | 骨格 |
|---|---|
| 単一タスク集中 | 中央1カラム。フォームは640px前後、読み物は720px前後 |
| Dashboard | `今やること` + 作業対象リスト + 補助一覧 |
| 詳細表示 | ヘッダー + 本文2カラム、またはタブ切替 |
| エディタ系 | 上部/左の文脈 + 中央編集面 + 必要な補助 |
| 一覧から詳細 | リスト + ドリルダウン、または選択行 + 右プレビュー |
| モバイル | 縦スタック + 下部固定CTA、または全画面Dialog/StepperDialog |

シフトリは店舗運営の作業アプリ。
SaaSの作業画面では、マーケ風の大きな装飾カードより、密度を保ったスキャンしやすい作業導線を優先する。
詳細は `references/layout.md`。

### 3. コンポーネント選定

まず既存の `src/components/ui/*` ラッパーと近いfeature実装を探す。
迷ったら `references/components.md` と `references/chakra-v3.md`。

- 汎用ボタン: `src/components/ui/Button` の `Button` / `IconButton`
- モーダル: `Dialog`
- 多段フロー: `StepperDialog`
- 空状態: `Empty`
- ローディング: まずSkeleton、必要なら `LoadingState` / `ShiftoriLoading`
- Select: `src/components/ui/Select`
- Toast: `toaster.create`
- Tooltip: `src/components/ui/tooltip`

薄いラッパーを増やす前に、その名前が業務上の意味・状態分岐・レイアウト責務を持つか確認する。
propsを流すだけなら既存UIを直接使う。

### 4. レイアウト・視覚階層

- 1画面1つの主役: 最優先アクションをサイズ・色・位置で明示する。残りはsecondary/ghostに落とす。
- F字・Z字スキャン: 左上から、重要情報、状態、アクションの順に置く。
- 近接・整列: 関係するものは近く、関係ないものは離す。縦軸か横軸で揃える。
- 8ptグリッド: 4/8/12/16/24/32/48/64 を基本にする。
- タイポ階層: `pageTitle` / `sectionTitle` / body系の既存textStyleを優先する。
- 色は意味: ブランド、状態、アクション、中立の意味にだけ使う。
- モバイル: PCの縮小ではなく、行をカード化する・ボタンを横幅いっぱいにする・全画面Dialogにするなど、指で操作できる構造にする。

### 5. 状態設計

以下を必ず見る。

- Loading: Skeletonが第一候補。Spinner/ロゴローディングはレイアウトが予測できない場合だけ。
- Empty: 初回、検索ゼロ件、処理済みで出し分ける。事実 + 次の一歩を書く。
- Error: フィールド、セクション、ページの3レベルを分ける。
- Success: Toast、永続バナー、インライン反映、画面遷移のどれで伝えるか選ぶ。
- Pending/Accepted: バックグラウンド処理は「受け付けた」と「完了した」を分ける。

詳細は `references/states.md`。

### 6. マイクロコピー

UI文言はこのスキル内で扱う。
ボタン、ラベル、エラー、空状態、Toast、確認ダイアログ、テーブル列、設定文言、スタッフ向け画面の文言をここで決める。
詳細ルールは下の「シフトリUI文言」。

### 7. 実装

- pages層で `useQuery`、ローディング、エラー、正常系の振り分けを行う。正常系のみfeaturesを呼ぶ。
- features層で `useMutation` とドメインUIを組む。
- フォームは react-hook-form + zodResolver。
- Submitは常にenabled。エラーは押下後に見せる。
- 二重送信はUIのdisabledだけに頼らず、`useSingleFlight` などのフロント同期ガードと、必要ならバックエンドの冪等性で止める。
- 日付はdayjs。`new Date().toISOString()` によるYYYY-MM-DD生成は禁止。
- SelectをDialog/StepperDialog内で使う場合は `usePortal={false}` を検討する。

### 8. 検証

- Storybookで代表状態を見る。
- 複雑な動きは play function で `expect(...)` と `findBy...` を使って検証する。
- `waitFor` は消滅やtransitionなど、`findBy...` で表現しにくい場合に限る。
- モバイルとデスクトップで、テキストの折り返し、CTAの位置、Dialogの高さを確認する。
- VRT対象にするか、振る舞いだけならsnapshotを切るかを判断する。

## 現状UIからのプロジェクト基準

### Dashboard

- Dashboardは `HeroSummary` の「今やること」を主導線にする。
- 送れなかった通知、スタッフ登録申請、締切/調整が必要なシフトは、独立したステータスページより `今やること` に集約する。
- 詳細・一括操作・行ごとの操作はモーダルへ逃がす。スマホでは全画面Dialogにする。
- 余計なサマリーバッジ、`すべて確認` のような曖昧な一括操作、関係ないフォームの巻き込みを増やさない。

### シフト一覧

- rawな日付順だけで並べず、ユーザーの作業意図でグルーピングする。
- 現在利用中、要シフト調整、募集中、確定済みのように、見れば次に何をするか分かる分類を使う。
- ユーザー向け表示は `要シフト調整`。内部名が `actionRequired` でも表面に `要対応` を出さない。
- 空グループは隠す。
- 過去シフトは初期表示に混ぜず、`過去のシフトを見る` / `もっと見る` で遅延表示する。
- 件数バッジは主張させすぎない。必要なら `blackAlpha.50` + `fg.muted` のような中立表現。

### 通知失敗・再送

- `通知エラー` で全部を潰さない。
  スケジューラー未設定、ジョブ未実行、enqueue失敗、outbox retry中、最終 delivery failure を分ける。
- 再送操作の成功は「再送を受け付けた」であり、メール/LINEが届いたことではない。
- 再送受付後の行は `retrying` / open list から除外する状態として扱う。
- 表示は `送れなかった通知`、`送れなかった相手を確認して、もう一度送れます。`、`再送受付済み` を基本にする。
- テーブル列は `スタッフ名`、`通知種別`、`募集期間`、`チャネル`、`検知日時` のように短く具体的にする。
- チャネルは `メール` / `LINE` と表示する。`mail` や内部チャネル名をそのまま出さない。
- 一括再送は強すぎる主役にしない。行ごとの確認と並存させる。

### ShiftForm / ShiftBoard

- 警告は編集中の邪魔をしない。
  日付ラベル/カードの件数バッジ、スタッフ名やセルの小さな警告アイコン/Tooltip、最終確認モーダルの要約を基本にする。
- 大きな黄色の警告枠でテーブルや確認モーダルを支配しない。
- 過去シフトなどサーバールールが絡む制約は、ボタンを隠すだけでなく、押下後に明確なToast/エラーで返し、バックエンドでも守る。
- `シフトを確定して通知` などの確定アクションは、警告がある場合にモーダルを開く前に検証し、同じエラー表示に合流させる。

### 設定・オンボーディング

- 初回登録の流れを基準にする。
  `日ごと` なら勤務時間ステップを出さない。
  `時間指定` なら勤務時間、`勤務区分` なら勤務区分を設定する。
- 店舗設定でも、初回登録と異なる独自ルールを急に作らない。
- StepperDialogでは、ステップ名を短くし、説明は現在の1ステップだけに置く。
- スタッフ登録やLINE連携など、誰に何が届くかは近くに説明する。

### スタッフ向け画面

- スタッフは管理画面の内部状態を知らない前提で書く。
- リンク不可や提出終了は、タスク単位で `このリンクでは提出できません` のように伝える。
- 次の行動は `シフト作成担当者に連絡してください` のように具体化する。
- 不安を下げるため、調整中・提出済み・受付終了の状態を簡潔に説明する。

## シフトリUI文言

忙しい店舗運営者とスタッフが、シフト作業を早く終えるための日本語を書く。
社内オペレーターやエンジニア向けの語彙に寄せない。

### 基本原則

- ふつうの現代的な日本語を使う。過度な敬語や役所っぽい語を避ける。
- 次に何をするか、何が起きたか、どの状態かを具体的に書く。
- ラベルは短く、補足はヘルパー、Dialog body、Alertに逃がす。
- 不安を下げるが、説明しすぎない。
- ユーザー語彙を使う: 店舗、店休日、募集、希望シフト、確定シフト、スタッフ、シフト作成担当者。
- 内部語を避ける: outbox、scheduler、retry、queue、token、magic link、ID、不達、非同期。
- バックグラウンド処理を過剰に約束しない。
  依頼を受け付けただけなら `受け付けました`。
  実際に送信予約/送信処理まで成功しているなら、コードで確認してから `送りました` / `通知しました` を使う。
- ユーザーが承認済みのローカル文言は、明示的な改善依頼がない限り尊重する。

### 文言を書く手順

1. Surfaceを決める。
   Manager app、staff-facing link、auth、landing/demo、operational notification、public pageのどれか。
2. 要素を決める。
   Button、label、helper、dialog、toast、empty、error、loading、table column、section heading、onboarding copyなど。
3. 最小の文言を書く。
   Labelは名詞句、Buttonは必要なら動詞 + 目的語、Bodyは1〜2文、Errorは何が起きたか + 次の行動。
4. プロダクトの事実を確認する。
   通知、削除、権限、課金、法務同意、配送/送信タイミングはコードを確認する。
5. Storybook play testや表示テキストのassertがあれば更新する。

### Buttons and CTAs

- 具体的に書く: `変更を保存`、`通知を確認`、`この募集を削除`、`確定シフトを送る`。
- `OK` は使わない。
- `保存`、`送信` だけで対象が曖昧なら目的語を足す。
- スタッフ向けは短くする: `提出する`、`もう一度開く`、`同意して続ける`。
- 破壊的操作はボタンだけでなくDialog title/bodyにも対象と結果を書く。

### Dialogs

- Titleは対象 + 行動/状態: `送れなかった通知`、`スタッフ登録申請`、`スタッフを削除`。
- Bodyは操作後に何が起きるかを書く。
- 破壊的bodyは不可逆性を直接書く。
  `本当に削除してよろしいですか？` より `この募集を削除すると元に戻せません。`
- Async bodyは downstream delivery の成功を断言しない。

### Toasts

- 短い結果文にする。
- バックグラウンド処理は `受け付けました` を優先する。
  例: `送れなかった通知の再送を受け付けました`。
- enqueue/schedule成功だけで `届きました` `完了しました` と書かない。
- 役に立つ場合は対象を入れる。
  例: `変更があるスタッフへの通知を受け付けました`。

### Empty States

- `事実 + 次の一歩`。
- `ありません` だけで終わらせない。
- 初回はCTAを置く。
  例: `期間と締切を決めて、スタッフに希望を聞きましょう。`
- 処理済みは安心させる。
  例: `もう一度送る必要がある通知はすべて処理済みです。`

### Errors

- 何が起きたか、どう回復するかを書く。
- ユーザーを責めない。
- `エラーが発生しました` だけで終わらせない。
- スタッフリンクでは `このリンクでは提出できません` のように、できないタスクを書く。
- 通信/端末問題は `再読み込み`、`少し時間をおく`、`Safari/Chrome/Edgeで開く` など具体行動を書く。
- フィールドエラーはその場で直せる文にする。
  例: `メールアドレスの形式が正しくありません。`

### Tables and Labels

- 短い名詞を使う: `スタッフ名`、`通知種別`、`募集期間`、`チャネル`、`検知日時`。
- 提出パターンは `日ごと`、`時間指定`、`勤務区分`。
- ヘッダーに文章を書かない。

### Preferred Terms

| Prefer | Avoid | Note |
|---|---|---|
| 送れなかった通知 | 不達通知 | `不達` は内部配送語に見える |
| 送れなかった相手を確認して、もう一度送れます。 | 送信できなかった通知を確認して再通知できます | 人と行動を出す |
| 再送受付済み | 再通知済み | 届いたとは限らない |
| スタッフ登録申請 | スタッフ参加申請 | 店舗運用では登録が自然 |
| スタッフ本人が登録できます | スタッフ自ら登録できます | 堅さを落とす |
| メールで届きます | メールにて届きます | `にて` を避ける |
| このリンクでは提出できません | このリンクは使用できません | タスクを明示する |
| 定休日があれば選択してください。休みはシフト募集時にも変更できます。 | お休みの曜日を押してください。また、募集ごとに細かく調整することも可能です。 | 幼い表現と `可能です` を避ける |
| LINE連携済みのスタッフにはLINE、それ以外のスタッフにはメール | LINE連携済みはLINE、未連携はメール | 本文なら前者、UI内の短文なら後者も可 |
| 提出内容をもとに、お店でシフトを調整しています。 | シフトは調整中です。 | スタッフ向けに安心材料を足す |
| 要シフト調整 | 要対応 | 業務意味を具体化する |

### Tone

- `できます`、`してください`、`届きます` の丁寧なアプリ日本語。
- `お願いいたします`、`よろしいですか`、`くださいませ` は避ける。
- Manager UIで `押してください`、`おまちください` は避ける。
- 名詞の連結より動詞を使う。
  `再通知の実行` より `もう一度送る`。
- 1文1アイデア。モバイルでは特に短くする。
- 句読点は本文に使う。タイトル・ボタンは基本的に句読点なし。

### 文言レビュー出力

既存文言レビューを求められたら、指定がなければこの表を使う。

| 優先度 | 画面 | 要素 | 現在の文言 | 違和感 | 改善案 |
|---|---|---|---|---|---|

優先度は `高 / 中 / 低`。
画面名はユーザーが分かる名前、要素は `Toast`、`Dialog title`、`CTAボタン`、`Empty state`、`説明文`、`テーブル列` などで書く。

## ユーザー指摘をスキルへ反映する

ユーザーからUIに関する指摘、好み、違和感、過去方針のリマインドを受けたら、このセクションを使う。

### 反映するもの

- 今後も使うUI判断。
  例: Dashboardの通知失敗は `今やること` に置く。
- 言葉の好み。
  例: `要対応` ではなく `要シフト調整`。
- 状態モデルの表現。
  例: 再送成功は配送完了ではなく `再送受付済み`。
- 視覚表現の傾向。
  例: 警告は大きい黄色枠ではなく、軽量なbadge/iconと確認モーダル要約。
- 検証・Storybook・VRTの習慣。

### 反映しないもの

- その画面だけの一時的な修正。
- 数値やPR番号など時間で変わる情報。
- まだユーザーが決めていない議論途中の案。
- コードで解決すべきバグを、曖昧なスキル文に逃がすこと。

### 更新手順

1. この `SKILL.md` と、必要なら該当referenceを読む。
2. メモリーを短く検索し、同じ指摘が過去にもあったか確認する。
3. 指摘を `ローカル修正`、`再利用可能ルール`、`文言用語`、`検証習慣` に分類する。
4. 再利用可能なら、このファイルの最も狭いセクションに1〜3行で追加する。
5. referenceに詳細が必要な場合だけ、該当referenceも更新する。
6. 既存ルールと衝突する場合は、より新しいユーザー指摘か、より具体的なプロダクト事実を優先する。
7. 最後にユーザーへ、どのルールを追加/調整したか日本語で伝える。

メモリー自体は、ユーザーが明示的に「メモリーに残して」と頼んだ場合だけ更新する。

## アンチパターン

- モーダル乱用: ページ遷移や既存導線で済むことをモーダルに詰める。
- ボタンが全部同じ重さ: プライマリは1つ。
- ラベルなしアイコン: 初見で分からないIconButtonには `aria-label` と必要ならTooltip。
- 空状態なし: 初回ユーザーが詰まる。
- スピナーだけ: 長い読み込みはSkeletonでレイアウトを守る。
- プレースホルダーをラベル代わりにする。
- モバイルで多カラムフォーム。
- 色の乱用: teal/orange/green/gray/redの意味を崩さない。
- 中央寄せ羅列: 作業UIは左寄せのほうが読みやすい。
- 枠線の二重化: section borderとcard borderが重なってノイズになるなら片方を消す。
- 破壊的確認の `本当に？`: 何を削除し、戻せないかを書く。
- 内部状態名の露出: `actionRequired`、`retrying`、`outbox`、`token` をそのまま出さない。
- 非同期の過剰約束: 受け付けただけなのに `完了しました` と書かない。
- 文言だけで業務ルールを守る: backend/date/permissionでも守る。

詳細は `references/anti-patterns.md`。

## プロジェクト固有ルール

### フレームワーク

- Chakra UI v3 + `src/components/ui/*` を優先する。
- 現状確認できる主要ラッパー: `Button` / `IconButton` / `Dialog` / `StepperDialog` / `Empty` / `FormCard` / `LoadingState` / `ShiftoriLoading` / `Select` / `Title` / `InfoGuide` / `LazyShow` / `Tour` / `toaster` / `tooltip`。
- `react-icons/lu` を既存パターンとして優先する。
- 専用BottomSheetラッパーは現状確認できない。モバイルの重いDialogは、既存実装に合わせて全画面Dialog/StepperDialogにする。

### 色

- `teal`: ブランド・主アクション。
- `orange`: 要シフト調整・警告・今すぐ見るべき状態。
- `green`: 募集中・達成・完了。
- `blue`: 確定済み・現在利用中など安定状態。
- `gray`: 中立・補助・完了済み・非アクティブ。
- `red`: 削除・致命エラー。
- `yellow`: 基本使わない。

### 状態管理・データ

- `selectedShopAtom`、`userAtom` など既存Jotai atomsを尊重する。
- Convex `useQuery` はpages層、`useMutation` はfeatures層。
- pagesでエラー/ローディング/正常を振り分ける。
- persisted shapeやConvex schemaが変わるなら、widen-migrate-narrowで移行を考える。

### Storybook / VRT

- `@storybook/react-vite` を使う。
- Storybook play testは `storybook/test` の `expect` を使う。
- `@storybook/test` パッケージはない。`fn()` は使わず `() => {}`。
- UI状態ごとのStoryを置く。VRTで撮るか、振る舞いだけならsnapshotを切るか判断する。
- fixed headerを含む縦長ページは `parameters.vrt.releaseFixedHeader = true`。
