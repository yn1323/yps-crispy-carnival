# ゴール分解：やりたいこと → UI

UI設計の90%はここで決まる。コードを書く前にユーザーの要求を以下の順で翻訳する。

## 1. Jobs-to-be-Done で捉える

「機能を作る」ではなく「ユーザーがどんな状況で、何を達成しようとしていて、そのあとどうなりたいか」で考える。

> **例**: 「シフト表を作る機能」ではなく、「店長が、月末の夜、疲れてる中、翌月のシフトを、スタッフの希望を踏まえて、30分以内に確定したい」

Jobsが決まると：
- 時間制約 → 入力を最小化、デフォルト活用
- 疲労状態 → 情報を詰め込まない、クリック回数を減らす
- 月1回 → 操作を覚えてない前提でラベルを丁寧に

## 2. タスクを3層に分ける

| 層 | 役割 | 視覚的扱い |
|---|---|---|
| **プライマリー** | この画面で一番やらせたい行動（1つだけ） | 最強の視覚的重み：サイズ・色・位置（右下 or ヘッダー） |
| **セカンダリー** | 補助的に取れる行動（3つまで） | secondary/ghostボタン、タブ、小さいラベル |
| **ターシャリー** | 偶に使うが常駐させる必要あり | Menu/Overflow menu (…)、設定画面に隠す |

**プライマリーが2つあると感じたら**：分割する。1画面1ゴール。

### 「セカンダリが多すぎる」の兆候

- トップに並ぶボタンが4つ以上
- タブとボタンが混在
- ページ下に「その他のアクション」がいっぱい

→ Overflow menu（…）に畳むか、サブページに追い出す。

## 3. ユーザーコンテキスト質問リスト

実装前に最低限これだけは言語化する：

- **誰が**: 役割・リテラシー・業務知識（店長 / スタッフ / 本部 / 初回ユーザー / リピーター）
- **頻度**: 毎日 / 週1 / 月1 / 年1 / 初回のみ → 頻度低いほど説明を丁寧に
- **デバイス**: PC / タブレット / スマホ / 横向き利用があるか → モバイル前提なら縦1カラム、親指ゾーン
- **時間圧**: 落ち着いて / 忙しい / 割り込み頻繁 → 忙しいほど操作を減らす
- **環境**: 明るい / 暗い / 片手 / 手袋 → 極端なコンテキストならタッチターゲット大
- **前提知識**: 業界用語わかる？データ構造理解してる？ → わからない前提でラベルを選ぶ

## 4. データ形状を決める

| 軸 | 選択肢 |
|---|---|
| 件数 | 1件 / 少数（〜10） / 多数（〜100） / 大量（100+） |
| 読み or 書き | 読み中心 / 書き中心 / 両方 |
| 更新頻度 | 静的 / 日次 / リアルタイム |
| 階層 | フラット / 親子 / ネット構造 |
| 属性数 | 1〜3 / 4〜10 / 10+ |
| 関係 | 独立 / 比較したい / 時系列 / 地理 |

これが決まると見せ方が決まる：

- 1件 × 多属性 × 読み中心 → 詳細ページ（左ラベル・右値のFormCard的配置）
- 多数 × 比較 → Table
- 多数 × 視覚 → Grid of cards
- 時系列 → Timeline / カレンダー
- リアルタイム → 差分表示、Toastで変化通知

## 5. フロー（画面遷移）を先に描く

コンポーネントを選ぶ前に、ユーザーの動線を箇条書きで書く：

```
1. ホーム → [予定] タブをタップ
2. 今週のシフト一覧表示
3. 行をタップ → 詳細 BottomSheet
4. [編集] → 編集フォーム BottomSheet に切替
5. [保存] → Toast 表示、BottomSheet閉じる、一覧に戻る
```

フローが3〜5ステップで完結しないなら、どこかに無駄がある：

- 毎回ホームに戻らせてない？
- 確認ダイアログ2連続になってない？
- 入力後にまた最初から表示されてない？

## 6. エッジケースの洗い出し

ハッピーパス以外に以下を最低限考える：

- **初回**（空状態、データなし）
- **ゼロ件**（検索・フィルタでヒットなし）
- **権限なし**（閲覧のみ / 編集不可）
- **長すぎる**（文字数・件数が想定外に多い場合のレイアウト崩れ）
- **オフライン・通信エラー**
- **重複・競合**（同時編集、バリデーション衝突）
- **破壊的操作後**（削除した直後の画面、undo有無）

**実装にはこれら全状態のStorybook Storyを作る**。

## 7. 判断を言語化する（ユーザーに共有）

実装前に短く決定事項をユーザーに伝える：

> **設計メモ**
> - プライマリー：「希望を登録する」→ 右下FAB
> - セカンダリー：先月コピー / 一括クリア → ヘッダーMenu
> - IA：センター1カラム（モバイルファースト、max 600px）
> - 主要コンポーネント：DatePicker + Select（時間帯）+ Textarea
> - 状態：loading=Skeleton、empty=初回ガイド、error=inline
> - プライマリーCTA=teal、削除=red

これでユーザーが方向性を承認できる。迷いが減る。

## 代表的な分解例

### 例1：「シフト希望を提出したい」

- **プライマリー**：希望日時の登録（多くてn件）
- **セカンダリー**：先月の希望コピー、不可日を一括指定
- **ユーザー**：スタッフ（Clerkアカウントなし・マジックリンク）、月1、スマホ、夜に短時間
- **データ**：日付×シフト時間帯の多次元、10〜30件
- **フロー**：マジックリンク → 日一覧 → 日タップ → BottomSheet選択 → 保存 → 次の日 or 完了
- **IA**：縦スクロール日付一覧 + BottomSheetで詳細入力
- **コンポーネント**：日付リスト（Card list） / BottomSheet / SegmentedControl（時間帯）/ Textarea（メモ）/ 送信ボタン（固定下部）
- **空状態**：「まだ希望が登録されていません。日付をタップして始めましょう」
- **エラー**：「保存に失敗しました。通信を確認してもう一度タップしてください」

### 例2：「今月の売上ダッシュボード」

- **プライマリー**：今月の状況を一目で把握
- **セカンダリー**：期間切替、ドリルダウン
- **ユーザー**：店長・本部、毎日、PC+モバイル、朝いち
- **データ**：日次集計、複数指標、前月比較
- **フロー**：ホーム → ダッシュ → 期間切替 or カードタップで詳細
- **IA**：上部KPIカード帯 + 中段推移グラフ + 下段詳細テーブル
- **コンポーネント**：KPI Card（数値+前月比バッジ） / LineChart / Table / DateRangePicker + プリセット
- **空状態**：「今月のデータはまだありません」
- **loading**：KPI/グラフ/テーブルそれぞれSkeleton

## やらないこと（爆発を防ぐ）

- **「このボタンも足しておく」**：プライマリーの格を下げる
- **「念のため」のオプション**：デフォルトを選び、カスタムは設定画面へ
- **「全部出しておく」**：Miller's Law（7±2）を超えたらカテゴリ化 or 隠す
- **「分析もできるようにしとく」**：プライマリーが薄まる → 別画面
