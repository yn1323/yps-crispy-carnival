# UIマイクロコピー（プロダクト内の短いテキスト）

ボタン、ラベル、ヘルプ、エラー、空状態など、画面内の短文を書く／直すときに参照。

マイクロコピーとは、UI上のごく短いテキスト（ボタン、フォームラベル、エラーメッセージ、ツールチップ、空状態など）のこと。**読み手が意識して読まなくても、行動を支える**ためのもの。1語変えただけでCVが数十％変わることがある（例：ボタン文言の変更で1.8倍、33％改善などの実績が報告されている）。

## ボタン

### 基本ルール

- **動詞で始める**
- **押した後に何が起きるかを書く**（結果志向）
- **2〜6語以内**（UIの横幅制限を意識）
- ボタン文言と、押したあとの画面の見出し・トーストは**一致させる**（期待の整合）

### ダメな例と直し方

| ❌ 曖昧 | ✅ 結果を書く |
|---|---|
| 送信 | 無料で相談する |
| 登録 | 無料でアカウントを作る |
| OK | 保存して閉じる／削除する |
| 続ける | シフトを確定する |
| 完了 | このシフトで確定 |
| 戻る | シフト一覧に戻る |

### ボタンの隣に添えるマイクロコピー

ボタン単体で伝えきれない「不安解消」「補足」は、**ボタンの直下か直横に1行**で添える。

```
[ 30秒で無料登録 ]
クレジットカード不要／いつでも解約OK
```

典型パターン：

- 所要時間：「1分で完了」「入力は3項目だけ」
- 不安解消：「クレカ不要」「いつでもキャンセルOK」
- 次の期待：「メールで確認リンクをお送りします」
- 金額の念押し：「無料のまま。課金は明示的に同意したときだけ」

### 破壊的操作の文言

削除・解約など「取り返しがつかない」系は、ボタン文言に**対象と行為**を両方入れて、誤タップを防ぐ。

- ❌「OK」「削除」
- ✅「このシフトを削除する」「アカウントを完全に削除する」

確認ダイアログでは：

- タイトル：何が起きるかを結果で書く（「このシフトを削除しますか？」）
- 本文：取り戻せないことを明示（「削除するとスタッフの希望も一緒に消えます」）
- ボタン：肯定＝行為動詞（「削除する」）／否定＝取り消し（「やめる」「キャンセル」）

## フォーム（ラベル・ヘルプ・プレースホルダー）

フォームはLPや機能紹介と並んで**離脱率が一番動くゾーン**。マイクロコピーで完了率が大きく変わる。

### ラベル（入力欄のタイトル）

- **必ず枠の外・上に固定**で置く（フォーカス時に消えない）
- 1〜3語で十分。文章にしない
- 必須／任意は**ラベルに併記**。記号より言葉（「（任意）」）で

```
名前（任意）
メールアドレス
電話番号
```

### プレースホルダー（枠の中の薄い文字）

- **ラベルの代わりに使わない**（フォーカスすると消えるのでユーザーが混乱する）
- 使うなら**入力例だけ**。形式を示すヒントとして使う

```
メールアドレス
[ example@shiftri.jp        ]
```

❌ プレースホルダーに「メールアドレスを入力してください」だけ入れる（ラベル代わり）
✅ ラベルをラベル欄に出し、プレースホルダーには形式の例だけ

### ヘルプテキスト（入力欄の下の補足）

- 入力が済まないと分からないルール、あるいは**不安解消**のための補足に使う
- 1行、30字以内が目安

```
パスワード
[                        ]
8文字以上 英数字と記号を組み合わせてください

メールアドレス
[                        ]
通知の受信に使います 公開されません
```

### 必須・任意の表現

- 少ない項目を目立たせる（多いほうは省略）
- 必須が多ければ「（任意）」を付ける／任意が多ければ「＊」や「必須」を付ける

### フォームのCTA

フォーム末尾のボタンは、フォームで得られるもの＝結果を書く。

| ❌ | ✅ |
|---|---|
| 送信 | 無料で資料を受け取る |
| 登録 | 無料で使い始める |
| 申し込む | 30秒で申し込む |

## エラーメッセージ

エラーはUXライティングで**最もユーザーを救える場所**。設計の粗さが一番出る場所でもある。

### 3つの要素（順に書く）

1. **何が起きたか**（事実を短く）
2. **なぜ起きたか**（必要なら）
3. **どうすれば直るか**（必ず書く）

短くするために全部入れなくてよい場合もあるが、**「どうすれば直るか」は絶対に抜かない**。

### 良い例・悪い例

| ❌ NG | ✅ 推奨 |
|---|---|
| 間違っています | メールアドレスに @ を入れてください |
| 認証に失敗しました | メールアドレスかパスワードが違います |
| 入力エラー | 電話番号は半角数字で入力してください |
| 不正なリクエスト | しばらくしてからもう一度お試しください |
| セッションが終了しました | しばらく操作がなかったため 自動でログアウトしました |
| システムエラーが発生しました | 通信が混み合っているようです 30秒ほどしてもう一度お試しください |

### 禁止ワード

- 「不正な」「失敗しました」「無効です」「エラーです」（責める／冷たい）
- 「〜できませんでした」（なぜできなかったか、次に何をすべきかがない）
- 「システムエラー」（読み手には解決できない情報）

### 置き場所と表示

- **エラーが発生した場所のすぐ近く**に出す（画面全体のトーストではなく、入力欄の直下）
- 入力中に連打するような早出しエラーは出さない（フィールドを離れてから判定する）
- **赤色＋アイコン＋短文**のセットで視認性を確保する
- 文字だけで伝えようとしない（色や枠線も使う）

### 致命的エラー（サービス停止、500 系）

- 冷たい「エラーが発生しました」は書かない
- 軽いユーモアで緊張を和らげる（ただし製品のトーン次第）
- できる次の行動を1つ示す（「しばらくしてから再読み込み」「問い合わせる」）

```
うまく表示できませんでした
少し時間をおいて もう一度お試しください

[ 再読み込み ]  [ サポートに問い合わせる ]
```

## 空状態（Empty State）

データがまだない画面は、**オンボーディングの絶好の場**。白紙で放置しない。

### 必ず書く3要素

1. **ここは何の画面か**（見出し）
2. **ここで何ができるか**（ベネフィット・価値）
3. **最初の一歩**（アクションボタン）

### 例

```
シフトがまだありません
スタッフから希望を集めて 最初のシフトを作りましょう

[ 希望を集める ]
```

```
通知はまだありません
スタッフが希望を出したり 変更があったときに ここに届きます
```

### NG 例

- 白紙
- 「データがありません」だけ（価値もアクションも伝わらない）
- 「Nothing here」など翻訳漏れ

## ツールチップ・ヘルプアイコン

- 画面の中で**どうしても補足が必要な項目**のみ。ツールチップは頼りすぎない
- 1文・30字以内。詳細はヘルプページへ
- マウスオーバーで出す場合、**モバイルで読めない**ことに注意。タップでも開ける実装にする

```
時給
[ ＿＿＿＿ ] ⓘ 割増や残業代の計算に使います
```

## ステータス表示（トースト・ダイアログ）

### 成功トースト

- 結果を **短く事実だけ**：「保存しました」「送信しました」
- ドヤらない（「ばっちり！」「完璧です！」はNG）
- 必要なら Undo を併設：「削除しました [ 元に戻す ]」

### 確認ダイアログ

- タイトル：結果（起きることを先に）
- 本文：詳細・取り返し有無
- ボタン：行為動詞で肯定／「やめる」「キャンセル」で否定

### ローディング・進行中

- **何を待っているかを書く**
- 長く待たせる可能性があれば目安時間を添える

| ❌ | ✅ |
|---|---|
| 読み込み中... | シフトを集めています |
| お待ちください | 保存中です 5秒ほどかかります |
| Loading | 最新情報を取りに行っています |

## 不安解消（リアシュアランス）

ユーザーが行動を止めるのは**不安**が原因。データで見ると、短いマイクロコピー1行で完了率が大きく変わる。

### 足すだけで効く定型

- **決済直前**：「お支払いは後で。クレジットカードの登録は不要です」
- **個人情報の入力欄**：「入力いただいたメールは通知にだけ使います。公開されません」
- **解約の不安**：「いつでも解約できます。違約金はかかりません」
- **変更可能性**：「あとから変更できます」「いつでも設定から変えられます」
- **共有されない保証**：「共有されません／公開されません」

### 配置場所

- 決定ボタン（購入・登録・削除）の**直下**が一番効く
- 入力欄の**直下**に添えて「このフィールドは怖くないよ」を伝える
- モーダルのタイトル直下にも入る

## オンボーディング（初めて使う人向けの文言）

### 1画面目（ようこそ）

- **サービスが提供する1つのベネフィット**を言い切る
- 「始める前にアカウント登録を」などの義務から入らない
- 次の一歩を1つだけ提示（Skip も許す）

### 空状態＋ゴーストデータ

オンボーディング直後の空状態では、「お試しデータを入れる」ボタンを出すのが効く（ユーザーが自分のデータを入れる前に、価値を体験できる）。

### ツアー（プロダクトツアー）のマイクロコピー

- ステップ数を表示（「3/7」など）
- 各ステップ、**1つだけ**を伝える（1画面1機能）
- 「わかった」ではなく「次へ」「触ってみる」のように行動を連想させる
- いつでもスキップできる逃げ道を用意

## 小さい画面（モバイル）での注意

- ボタン文字は2〜4語（横に入らない）
- トーストは1〜2行
- 入力欄ラベルは**上に固定**（フォーカスで消すタイプはモバイルで特に不便）

## マイクロコピーの効果を上げる言葉の道具箱

使いすぎ注意だが、正しく使えば効く。

| 効能 | 言葉 |
|---|---|
| 時間短縮 | 「1分で」「30秒で」 |
| 無料感 | 「無料で試す」「無料のまま使える」 |
| 不要 | 「クレカ不要」「住所の入力なし」 |
| 変更可 | 「あとから変えられる」「いつでも解約OK」 |
| 所有感 | 「あなたのお店に合わせて」「自分だけの〜」 |
| 限定 | 「今だけ」「先着〇〇」※嘘にならないときだけ |
| 限定残 | 「残り3枠」※嘘にならないときだけ |

嘘・誇張は**一度ミスすると信頼が崩壊する**。使う前に、書いてあることを証明できるか確認する。
