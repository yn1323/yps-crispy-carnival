# 状態設計：Loading / Empty / Error / Success

UIに必須の4状態。Storybook Storyで全部見える状態を必ず用意する。

## Loading

### 第一候補：Skeleton

- レイアウト形状を保持（背景に薄いブロック）
- shimmer animation で動きを示す
- データが入る場所のサイズ・位置を予告

```
┌───────────────┐
│ ▓▓▓▓▓▓        │  ← タイトル相当
│ ▓▓▓▓▓▓▓▓▓▓▓▓  │  ← 本文相当
│ ▓▓▓▓▓▓▓▓      │
└───────────────┘
```

### Spinnerは最後の手段

使うべきタイミング：
- 操作直後の短時間（0.5〜2秒、ボタン内 spinner）
- レイアウトが予測できないコンテンツ（小さなウィジェット）
- ファイルアップロード等の進捗不明

避けるべきタイミング：
- 長時間（10秒以上）：Spinnerを見続けるのは苦痛 → Skeleton or progress
- ページ全体（FullPageSpinner濫用）：白画面が長い → Skeleton

### ボタン内 Loading
- ボタン disable + 内部 spinner
- ラベルを"処理中..."に変える、または変えずアイコンのみ
- 二重押下防止が主目的

### スケルトンの粒度
- 細かすぎ：すべての要素を skeleton 化 → 結局見づらい
- 適切：主要なテキストブロック・カード・画像のみ
- 行数は実データの平均件数（10件相当）でOK

### Suspense / Streaming パターン

- ページ全体をブロックせず、セクションごとに段階的に出す
- 上部（KPI）→ 中央（グラフ）→ 下部（詳細）の順で読み込み

## Empty State

### 3種類を出し分ける

#### ① 初回（データがまだない）
- **必須**：イラスト or アイコン + タイトル + 説明 + Primary CTA
- 「最初の◯◯を作りましょう」「ここに登録するとXができます」
- 例: 「シフトをまだ登録していません。日付をタップして始めましょう」

#### ② ゼロ件（フィルタ・検索後）
- 検索条件の表示
- 「条件を緩める」「クリア」のCTA
- 例: 「『山田』に該当するスタッフはいません。条件をクリアしてみてください」

#### ③ クリア済み（データを全部消した後）
- ミニマル：「データがありません」のみ
- 必要なら「再度作る」CTA

### 鉄則
- 装飾だけの Empty（"Empty"とだけ書く）は禁止
- ユーザーの次の行動を必ず提示
- イラストはシステム内で統一トーン（バラバラだと安っぽい）
- 文言詳細 → `/ux-writing`

このプロジェクトでは `src/components/ui/Empty` ラッパーを利用。

## Error State

### 3レベルで出し分ける

#### フィールドレベル
- 該当フィールド直下に赤字
- 該当フィールドに赤枠
- 何をどう直すかを書く（"半角英数で入力してください"）

#### セクション/カードレベル
- セクション内のデータ取得失敗
- カード内に「読み込めませんでした [再試行]」
- 周りの操作はブロックしない

#### ページレベル
- 致命的・回復不能 or リトライ必要
- 全画面エラー：何が起きた + 次の行動
- 「ホームに戻る」「再読み込み」「サポートに連絡」

### Error コピー（基本）

「何が・なぜ・次どうする」

❌ "An error occurred"
✅ "通信が不安定です。少し時間をおいてもう一度お試しください"

❌ "Failed to load"
✅ "シフトを取得できませんでした。[再試行]"

### 404 / 403 / 500

- **404**：「ページが見つかりません。URLが正しいかご確認ください。[ホームへ]」
- **403**：「このページの閲覧権限がありません。管理者にお問い合わせください」
- **500**：「サーバーで問題が発生しました。しばらくしてもう一度お試しください」

### Error Boundary

このプロジェクトでは `src/components/ui/ErrorBoundary` を活用。レンダリング例外をキャッチしてフォールバックUIを出す。

## Success State

### パターン
- **画面遷移**：完了画面 / 一覧に戻る + Toast
- **インライン**：チェックマーク + 「保存しました」（数秒後消える）
- **Toast**：別画面なら、操作直後に右上 / 下部
- **大きな成功**（決済完了、登録完了等）：専用画面 + サマリ + 次のCTA

### 出さないことも考える
- 即座に変化が見える（Switch ON でON状態になる）→ Toast不要
- ページ遷移したらそれが成功の証 → Toast出さない選択もあり

### 過剰な祝福を避ける
- 紙吹雪・大きなアニメーションは初回登録などのマイルストーンのみ
- 日常操作は静かに完了

## 楽観的更新（Optimistic Updates）

### 使うべき時
- ネットが遅くてもUIが軽快に見せたい
- 失敗確率が低い（権限OK、シンプルな更新）
- 失敗時の rollback が安全

### 実装パターン
1. UIを即座に更新（ローカル state を変える）
2. mutation 発行
3. 成功 → 確定（特にUI変更なし）
4. 失敗 → ロールバック + 「保存に失敗しました [再試行]」 Toast

### 危険な楽観更新
- 課金・決済（必ず明示確定）
- 削除（undoがないなら確認）
- 他ユーザーへの可視影響（投稿・送信）

## Loading × Empty × Error の優先順位

データ取得の3状態：

```
isLoading → Skeleton表示
isError → Errorメッセージ
data && data.length === 0 → Empty
data && data.length > 0 → コンテンツ
```

**順番が大事**：エラーがあるのに Skeleton 表示し続けると永遠に待たされる。

このプロジェクトの規約：**pages層で振り分け**、正常系のみ features/* に渡す。

## Refetch / Refresh

- 手動リフレッシュ：右上にリフレッシュアイコン
- Pull to Refresh（モバイル）：縦スクロール一覧で
- バックグラウンドリフレッシュ：データが古くなったら静かに更新（loading出さない or sublte indicator）
- リアルタイム購読（Convex useQuery 等）：自動更新、変化通知（Toast or バッジ）

## オフライン状態

- Banner で「現在オフラインです。再接続しています…」
- 編集はキューイングし、再接続後に送信
- 楽観更新と組み合わせると体感が良い
- 完全に保存前のデータは`localStorage` に下書きを保持

## ストーリーテリング

Storybook Stories（必須）：
- `Default` - 通常データ
- `Loading` - Skeleton
- `Empty` - 空（3種類別Storyにしてもよい）
- `Error` - エラー
- `Many` - 大量データ
- `LongText` - 文字溢れ

これでVRTで状態ごとに見え方をチェックできる。
