# Laws of UX：判断の根拠になる原則

UI判断の理由づけに使える主要な認知・行動原則。「なぜ」を説明するときに引用する。

## Fitts's Law（フィッツの法則）

**ターゲット獲得時間は距離と大きさで決まる**：遠い・小さいほど押しにくい。

### 適用
- 主要操作は大きく（最低 44×44px、デスクトップでも 32+）
- よく使う操作は近くに（モバイル下半分、デスクトップは現在のフォーカス周辺）
- 画面端は無限ターゲット（端なら超越できないので大きいのと同義）→ 主要メニュー・閉じる×を端に
- 危険な操作は遠く・小さく（誤クリック防止）

## Hick's Law（ヒックの法則）

**選択肢が増えるほど決定時間が長くなる**（対数的に）。

### 適用
- 選択肢を絞る、カテゴリ化（一度に7±2）
- メニューを階層化（最初は5項目、必要なら掘る）
- デフォルト選択を提供（多くは選ばずに通過する）
- 推奨をハイライト（"おすすめ"バッジ）

## Miller's Law（ミラーの法則・7±2）

**短期記憶に保てる項目は7±2個**。

### 適用
- 1画面に並べるカード・KPIは7以下
- ナビ項目は5〜7個まで
- フォームのステップ数は5〜7
- 多すぎる場合：チャンキング（カテゴリ・グループ化）

## Jakob's Law（ヤコブの法則）

**ユーザーは他のサイトで過ごす時間の方が長い**。普及した慣習に従えば学習コスト下がる。

### 適用
- 検索：虫眼鏡アイコン、画面上部 or サイドバー
- 戻る：左上
- ハンバーガー（モバイル）：左上 or 右上
- ロゴ：左上、クリックでホーム
- カート/通知/プロフィール：右上
- 主要CTA：右下（フォーム）or 上部
- 慣習を壊すなら、相応の理由とフィードバックが必要

## Doherty Threshold

**応答時間が400ms以下なら生産性が劇的に上がる**。

### 適用
- インタラクションのフィードバックは即座に（楽観的更新、Skeleton即表示）
- 検索debounce 250〜400ms（即時すぎず、遅すぎず）
- 重い処理 → 進捗表示 + ETA（待っていることを明示）

## Aesthetic-Usability Effect

**美しいUIは使いやすく感じられる**（実際の使いやすさが同じでも）。

### 適用
- 視覚的な完成度に投資する価値はある（特にOnboarding・最初の画面）
- ただし「装飾」と「使いやすさ」は別軸：両立を狙う
- 装飾過多 → 機能性を犠牲にしているならNG

## Peak-End Rule

**体験は「ピーク」と「終わり」で記憶される**。

### 適用
- 完了画面に投資（決済完了、登録完了の祝福）
- エラーからのリカバリー体験を磨く（エラーが綺麗で次の手が明確だと印象良い）
- ログアウト・退会フロー：失礼にならず簡潔に

## Tesler's Law (Conservation of Complexity)

**複雑さは消えない、誰かが引き受ける（システム or ユーザー）**。

### 適用
- ユーザーに見える複雑さを減らす（バリデーション・整形を自動で）
- スマートデフォルト、自動入力（住所からの郵便番号予測）
- 「設定が必要」より「動いてからカスタマイズ」

## Postel's Law (Robustness Principle)

**入力には寛容に、出力には厳格に**。

### 適用
- 電話番号：ハイフン有無どちらでも受け取る
- 日付：複数形式を解釈
- メール：前後空白を trim
- ただし、表示・保存形式は統一

## Goal-Gradient Effect

**目標に近づくほど加速する**。

### 適用
- 進捗バー（"あと2ステップ"）
- 達成可能性を可視化（"60% 完了"）
- 多段フォーム冒頭は短くする（最初の1歩を軽く）

## Von Restorff Effect (Isolation Effect)

**周囲と異なるものは記憶される**。

### 適用
- プライマリーCTAは色・大きさで突出させる
- 重要な通知は他と違う色/形
- 強調しすぎは禁物（突出が複数 = 突出が消える）

## Gestalt 原則

### 近接（Proximity）
近くにあるものは関係していると認識される。
→ 関連項目を近づけ、無関係は離す。

### 類似（Similarity）
似た見た目は同じ意味と認識される。
→ 同じ役割は同じスタイル、異なる役割は異なるスタイル。

### 連続（Continuity）
線・曲線で繋がるものは1グループ。
→ 表の罫線、リストの divider。

### 閉合（Closure）
閉じた図形を1まとまりと見る。
→ カードの枠線、グループの背景。

### 共通運命（Common Fate）
同じ方向に動くものは1グループ。
→ ホバーで一斉に明るくなるカード = グループ。

### Figure-Ground
前景と背景を区別する。
→ Modalは前景、暗くした overlay は背景。

## Serial Position Effect

**リストの最初と最後が記憶されやすい**（中間は弱い）。

### 適用
- ナビゲーション：重要項目を最初 or 最後
- メニュー：最も使うものを上、ログアウト等は下
- 検索結果：上位を最適化

## Zeigarnik Effect

**未完了タスクは記憶に残る**。

### 適用
- "あなたのプロフィールが80%完了です"
- TODOチェックリスト
- 入力進捗のフィードバック

## Cognitive Load（認知負荷）

3種類：
- **内在的**：タスク自体の難しさ
- **外在的**：UIが課す不必要な負荷（ここを最小化）
- **学習関連**：理解・スキル向上に使う負荷（適度に）

### 適用
- 不必要な装飾・選択肢を削る
- ラベルを明示（記憶ではなく認識で）
- スマートデフォルト
- プレビュー・即時反映

## Recognition over Recall

**再認は再生より楽**。

### 適用
- 入力候補を提示（Combobox、autocomplete）
- 過去入力を覚えさせる
- アイコン+ラベル併記
- 操作のヒントを文脈に置く（プレースホルダー、Tooltip）

## Pareto Principle（80/20）

**20%の機能が80%のユースケースをカバー**。

### 適用
- メイン操作にUIリソースを集中
- 高度な機能は隠す（Settings、Advanced）
- 統計を見て使われない機能はリストラ

## Banner Blindness

**バナー風の要素はユーザーが無視する**（広告慣れ）。

### 適用
- 重要通知をバナー風（横長・背景色）にしない
- インラインメッセージ、Toastなど形を変える
- スティッキーで大きいバナーは特に無視される

## このプロジェクトでの適用例

- **シフト一覧**：Hick's → 1日の選択肢を絞る、Miller's → 週単位で7日
- **ダッシュボード**：Serial Position → 重要KPIを左上、Pareto → 上部4つで主要指標
- **フォーム**：Doherty → onBlurバリデーション即応、Recognition → 候補補助
- **空状態**：Goal-Gradient → 「最初の1件を作りましょう」（1歩を軽く）
- **エラー**：Peak-End → エラー画面のクオリティを落とさない
- **モバイル**：Fitts's → 下部固定CTA、ヘッダー×を端に
