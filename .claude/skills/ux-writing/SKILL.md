# UXライティング（日本語・Web向け）

## このスキルでやること

Webページ・SaaS・LPに載るあらゆる日本語テキストを、**読み手のわかりやすさ最優先**で書く。技術者・マーケターが書きがちな「難しい・長い・他人事」な文章を、「短い・具体的・自分ごと」に置き換える。

対象テキストの例：

- **LP**: ヒーロー（H1/サブヘッド）、ベネフィット見出し、特徴説明、社会的証明、料金、FAQ、CTA
- **プロダクトUI**: ボタン、フォームラベル／ヘルプ／プレースホルダー、トーストやダイアログ、エラーメッセージ、空状態、ツールチップ、オンボーディングの文言、設定画面の説明
- **ドキュメント／お知らせ**: ヘルプ記事の見出し、変更通知メール、アプリ内のアナウンス

このスキルは**汎用**。特定ドメインに縛られないが、本ワーキングディレクトリ（シフト管理SaaS）のような**小規模店舗・個人事業主向け**プロダクトで特に効くよう、語彙やトーンを合わせている。

## 起動時に必ずやること

1. 文章を書く前に「**誰が読むのか／何の場面か／次に何をさせたいか**」を1文で言語化する。曖昧なまま書き始めない
2. CLAUDE.md の「テキストガイドライン」があれば**必ず優先**する（プロジェクト固有のトーンがある）
3. 下の「9つの原則」で判断し、細部は `references/` を参照する
4. 書き上げたら `references/checklist.md` の自己チェックを1周通してから返す

## 9つの原則（判断の根っこ）

判断に迷ったら上から順に適用する。上の原則が下の原則より優先。

### 1. 明確さ ＞ 美しさ ＞ 面白さ

読み手が**一瞬で同じ意味に解釈**できる言葉を選ぶ。曖昧な形容詞、多義的な熟語、かっこつけた言い回しを、具体で置き換える。

- ❌「充実したサポート体制」→ ✅「専任担当が平日9〜19時に対応」
- ❌「シンプルで使いやすい」→ ✅「最初の1日は登録だけでOK」

### 2. ベネフィット ＞ 機能

書き手が何を「作ったか」ではなく、読み手が何を「得られるか／できるようになるか」を書く。

- ❌「AIによるシフト自動生成機能」→ ✅「30人分のシフトが3分で組める」
- ❌「リアルタイム同期対応」→ ✅「スマホで変更するとレジ側もすぐ切り替わる」

### 3. 読み手を主語にする（"あなた"視点）

文の主語は原則「あなた／お店／スタッフ」。「当社は／弊社が」は避ける。話し言葉に近づける。

- ❌「当社の〜機能は〜を可能にします」→ ✅「〜が〜できるようになります」

### 4. 具体 ＞ 抽象

数字、固有名詞、動作で書く。「ぱっと／かんたん／スマート／サクサク」など擬音・擬態語は使わない（抽象で曖昧になる）。

- ❌「かんたんに登録」→ ✅「メールだけで30秒」
- ❌「多くの店舗が導入」→ ✅「700店舗で使われています」

### 5. 肯定形 ＞ 否定形／禁止形

「〜しないと」より「〜すれば」。ユーザーを責めない。

- ❌「支払わないと使えなくなります」→ ✅「続けてお使いいただくにはお支払いをお願いします」
- ❌「入力が間違っています」→ ✅「メールアドレスに@を入れてください」

### 6. 短く／1文1メッセージ

本文は1文80字以内が目安（日本語は50字でも読みにくい）。1文に情報を詰め込まない。並列は箇条書きへ。

### 7. 一貫性

同じ機能・同じ画面では**同じ用語・同じボイス**を使う。「スタッフ／従業員／メンバー」のブレ、「保存／登録／確定」のブレは読み手を混乱させる。

### 8. 誠実さ（盛らない）

実態と違う「自動で／勝手に／完璧に」は書かない。機能上の制約（例：通知はメール）も素直に書いて、価値として「手段」ではなく「体験」を訴える。

### 9. 次の行動を示す

読んだあとに「で、どうすれば？」を残さない。説明には必ず「次にできること」を添える。エラーには必ず「解決の一手」を書く。

## 書くときの定型フロー

どんなテキストでも以下の順で考える。

1. **役割の特定**: 見出し／本文／ラベル／ボタン／エラー／空状態 のどれか？
2. **読み手と状態**: 初見？ログイン済み？困っている？急いでいる？
3. **伝えたい1メッセージ**: 「この文を読み終えた後、読み手は何を知るか／何をすべきか」を1文で書く
4. **一次ドラフト**: 9原則に照らして書く
5. **削る**: 不要語（「こと」「もの」「における」「〜することができる」「〜いただきます」）を削る。2割削れる余地がほぼある
6. **日本語の形式**: `references/japanese-style.md` のルールを適用（漢字比率、句読点、数字表記、敬語）
7. **セルフチェック**: `references/checklist.md` のチェックリストで1周

## 場面別クイックリファレンス

判断の入口だけ。詳細は各 references を読む。

### LP（ランディングページ）

- **H1（ヒーロー）**: 誰が／何を得られるか を 15字前後で。ベネフィット1本化
- **サブヘッド**: H1だけで伝わらない「どうやって」「どれくらい」を1行で補足
- **本文**: 特徴→ベネフィット変換、社会的証明を添える
- **CTA**: 動詞で始め、結果を書く。「送信する」ではなく「無料で相談する」
- **不安解消**: CTA直下に「クレカ不要／いつでもキャンセル／30秒で完了」等を添える

→ 詳細: `references/lp-copy.md`

### UIマイクロコピー（プロダクト内の短い文）

- **ボタン**: 動詞始まり、**結果を書く**。「保存」より「シフトを確定する」
- **フォームラベル**: 必ず枠の外に固定。プレースホルダーにラベルを入れない
- **プレースホルダー**: 入力例のみ（例: `example@shiftri.jp`）
- **エラー**: ①何が起きた ②なぜ ③どうすれば を短く
- **空状態**: そこで「何ができるか」と「最初の一歩」を書く
- **ツールチップ**: 1文で。詳細はヘルプに逃がす

→ 詳細: `references/microcopy.md`

### 日本語の書式

- **見出し／サブタイトル**: 句読点なし、改行でリズム（体言止めOK）
- **本文**: 句読点あり、述語終わりで書く（「〜できます。」）
- **漢字：ひらがな ≒ 3:7**。漢字密度が高いと難解に見える
- **敬語**: ボタン・見出しは無敬語／本文は「です・ます」で統一。過剰な「お／ご／〜させていただく」を削る
- **数字**: 横書きは半角算用数字（2026年4月、3分、700店舗）
- **不用意な半角スペースを本文に入れない**

→ 詳細: `references/japanese-style.md`

### 小規模店舗・個人事業主ユーザー向け

- カタカナ用語を警戒する（「ダッシュボード」「アラート」「インテグレーション」を避け、「画面」「お知らせ」「連携」に）
- テック文脈の比喩（「AIが／自動化／最適化」）より、**具体の業務動作**（「シフト表が作れる」「スタッフに一斉送信」）で語る
- 「小さなお店」「零細」ではなく**「少人数のお店」「個人のお店」**のように前向きな表現

→ 詳細: `references/japanese-style.md` の「語彙の感覚」セクション

## references の使い分け

- **`references/lp-copy.md`**: LP／紹介ページ／料金／FAQ／訴求文を書くとき
- **`references/microcopy.md`**: プロダクトUI内の短文を書く／直すとき
- **`references/japanese-style.md`**: 日本語のトーン・文字種・句読点・敬語・用語選びで迷ったとき
- **`references/checklist.md`**: 書き終えた直後に必ず1周。自己レビュー用

## やってはいけない（NG パターン）

書いてしまったら必ず書き直すべきもの。

- **AI-slop な修飾語**: 「革新的な／シームレスな／最先端の／業界をリード／スマートに／ぱっと／サクサク」
- **自画自賛**: 「自信を持ってお届けします」「お客様に寄り添い続けてきた」
- **機能名の直訳**: 「当アプリはマルチデバイス対応」→「スマホ・PC・タブレットから使えます」
- **"〜することができます"**: ほぼ全部「〜できます」で置換できる
- **"〜していただきます"連発**: 敬意がインフレしてかえって距離ができる
- **「はじめに」だけ書かれた導入文**: 何も伝えていない。削る
- **同じことを別セクションで繰り返す**: 情報は削る方向で設計する

## 仕上がりの水準

以下が満たせていれば「出してよい」。

1. 1文ずつ声に出して読んだとき、一度も詰まらず意味が通る
2. 削れる語が見当たらない（削るとニュアンスが壊れる）
3. ユーザーが読んだあとに「で、次は？」と迷わない
4. 嘘・誇張・手段の押しつけが混ざっていない
5. 同じ機能を別の語で呼んでいない（一貫している）
